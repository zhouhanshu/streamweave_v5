# StreamWeave V4 代码重构记录

> 本文件用于记录 V3 中暴露的问题、V4 的重构目标，以及后续每轮代码修改需要关注的边界。
> `streamweave_v3` 只作为只读参考；所有新代码和重构产物都写入 `streamweave_v4`。

## 1. V3 当前定位

V3 已经跑通了新版单步协议的基础闭环：

```xml
<eta>...</eta>
<answer>...</answer>
<bridge t="...">...</bridge>
<note t="..." frame="N"/>
```

核心流程是：

```text
video_io.load_video_chunks
  -> rollout.RolloutRunner
  -> prompts.TeacherPromptBuilder / ProductionPromptBuilder
  -> backend.generate_content
  -> parser.parse_action_result
  -> action_quality.score_action_quality
  -> action_postprocess.InferencePostprocessor
  -> memory.MemoryStore
  -> trace / benchmark scoring
```

V3 的价值是提供了可运行 baseline、OVO/StreamingBench adapter、数据合成管线和 SFT 导出雏形。V4 的任务不是在 V3 上继续 patch，而是把这些能力重新整理成更清晰、更适合 SFT/RAFT/RL 的代码结构。

## 2. V3 已知问题

### 2.1 推理环境和训练环境耦合不清

- `RolloutRunner` 负责视频加载、query 注入、prompt 构造、模型调用、解析、质量评估、后处理、memory 更新和 trace 回调，职责过重。
- RL 需要 step-level environment API：`reset / build_observation / step(action) / trace`，而 V3 只有整条 rollout runner。
- verl 侧需要每一步 fresh prompt，而不是拼接多轮 transcript；V3 还没有独立的 `rl_env.py` 或 transition 数据结构。

### 2.2 raw quality、postprocess、reward 的边界需要更硬

- V3 已经区分 raw quality 和 inference postprocess，但训练/奖励代码还没有成体系。
- 推理可以修复 note 时间、bridge interval、open-tail bridge；但 SFT/RAFT/RL reward 应优先按 raw output 打分，避免模型靠 postprocess 掩盖协议错误。
- `TeacherPostprocessor` 尚未实现，教师数据过滤目前缺少专门策略。

### 2.3 Memory 实现重复

- 主推理使用 `streamweave.memory.MemoryStore`。
- SFT 使用 `data/sft/rollout_sft.py` 里的 `SFTMemoryState`。
- 两者 open-tail bridge、note eviction、memory rendering 逻辑高度相似，后续修改协议时容易不同步。

### 2.4 policy 只是接口，不是完整实验能力

- `gated` 当前等价于 full note+bridge 读取，没有真实 gate。
- `simplestream*` 当前只是禁用长期 memory，并没有实现真实 last-N frame/window cropping。
- 后续消融需要把 policy 从简单读写开关升级为可组合的 observation/memory rendering 策略。

### 2.5 prompt 和配置中存在实验临时代码痕迹

- `prompt_type=teacher` 在 eval 配置中广泛使用，实际线上/学生评估需要更短的 production prompt。
- 配置包含大量本机绝对路径。
- `configs/eval_ovo_qwen3vl32b_one.yaml` 中有明文 API key，V4 不应复制这种配置方式。

### 2.6 评测和结果统计还不完整

- OVO adapter 已覆盖 backward / realtime / forward，并修复了 forward 子视频 `{id}_{i}.mp4` 的加载逻辑。
- StreamingBench proactive 目前能加载和 rollout，但评分仍是 `score=None`，缺少 judge 或等价规则。
- batch eval 会把样本级异常写入 `error` 并给 `score=0`，但错误率、有效样本得分和 error-as-zero 得分需要更清晰地区分。

### 2.7 数据合成和 SFT 管线可以跑，但结构还偏脚本化

- `data/synthesize` 是五段式脚本管线：windows -> captions -> QA -> filter -> export。
- `data/sft` 复刻了一套 rollout 逻辑来导出 step-level SFT，中间表示有价值，但和主 runtime 的环境代码没有统一。
- 后续需要把 SFT/RAFT/RL 共享的 step transition schema 抽出来。

## 3. 最新需求拆解（2026-04-30）

下面按当前新需求整理 V4 需要重点解决的问题，并标注 V3 中对应的代码依据。

### 3.1 后端层：API、本地模型、多模型并行和内部重试

V3 现状：

- `streamweave/backends.py` 里 `OpenAICompatibleBackend.generate_content` 和 `GeminiBackend.generate_content` 都是单次请求接口。
- `evaluation/eval_batch.py` 的 `_worker_run` 在样本级做 retry，会重新跑整条 sample。这个机制可以兜底，但不适合处理后端瞬时失败。
- 后端返回值只有文本，缺少 endpoint id、attempts、latency、error trace、token usage 等运行指标。

V4 要求：

- 后端接口要能同时适配 API 服务、本地部署模型、OpenAI-compatible 服务、Gemini/Vertex 等。
- 支持 OpenAI-compatible/vLLM 的多个 endpoint 场景，例如本地 8B 模型按 8 张卡启动 8 个 endpoint，每个 endpoint 有独立并发上限。
- Gemini/API 后端不按 8 个 endpoint 设计，直接用同一个后端配置做并发 worker。V3 的 `evaluation/eval_batch.py` 已经是这种模式：`batch_ovo_gemini*.yaml` 里 `endpoints: []`，worker 会使用同一个 Gemini backend；只有本地 vLLM 多副本才需要 endpoint 列表。
- retry 放在 backend 内部：连接错误、限流、超时、临时 5xx 等应由 backend 根据 `RetryPolicy` 重试，并记录每次失败原因。
- batch 层可以保留 sample-level retry，但语义只作为整条样本灾难性失败兜底，不能替代 backend request retry。
- `generate` 返回结构化 `BackendResult`，至少包含 `text`、`latency_seconds`、`endpoint_id`、`attempt_count`、`retry_errors`，后续可扩展 `usage`。

### 3.2 分辨率：`resolution=0` 使用原始分辨率

V3 现状：

- `streamweave/backends.py` 的 `_resize_image(image, max_side)` 会按 `max_side` 缩放图片。
- 现在没有明确语义说明 `max_side=0`。如果直接传 0，`thumbnail((0, 0))` 这类逻辑会导致错误或无效图像。

V4 要求：

- 配置里保留 `resolution` 或 `max_image_side` 字段。
- `resolution == 0` 明确表示不缩放，使用原始分辨率。
- `resolution > 0` 表示最长边缩放到该值以内。
- 这条规则要写单元测试，并保证 API backend 和本地 backend 共用同一套图像预处理逻辑。
- 抽帧后的时间戳不强制依赖额外 metadata。按连续抽帧约定，给定 `sample_fps=f` 和 `frame_id_base=b`，文件编号 `frame_id` 对应时间区间 `[(frame_id-b)/f, (frame_id-b+1)/f]`。当前 1fps 数据下，第一张就是 `0-1s`。V3 的 `data/sft/frame_dataset.py` 已经按 `start = frame_id - frame_id_base`、`end = start + 1.0` 处理 1fps 帧。

### 3.3 代码结构：步骤必须解耦

V3 现状：

- `streamweave/rollout.py` 的 `RolloutRunner.run_chunks` 同时负责 query 注入、memory 更新、prompt 构造、backend 调用、parser、quality、postprocess、trace 回调。
- 这导致 eval、SFT 导出、后续 verl rollout 都容易复制一份 runner 逻辑。

V4 要求：

- 拆出真正的环境层 `StreamWeaveEnv`，只负责状态转移和 transition 记录。
- 模型调用、数据读取、batch 调度、prompt profile、postprocess profile、trace writer 都要独立。
- 主流程应能清晰表达为：

```text
sample -> frame store -> env.reset
while not done:
  prompt = env.build_prompt()
  backend_result = backend.generate(prompt)
  transition = env.step(backend_result.text)
  trace.write(transition)
```

### 3.4 一个框架服务三类任务

V4 同一套核心 runtime 需要支持：

- 教师模型评测跑分：关注最终答案、协议质量、动作是否可被修复并执行。
- 教师模型 SFT 数据合成：关注 raw 输出是否合格，错误时给提示重试，不修改模型原始输出。
- verl rollout：关注 step-level prompt/action/reward，不能拼接多轮上下文，只能通过 memory 和 QA history 传递状态。

核心判断：这三类任务不应该各写一套 rollout。差异应只体现在 prompt profile、quality/reward profile、postprocess profile、output exporter。

### 3.5 三套 prompt profile

V3 现状：

- `streamweave/prompts.py` 只有 `TeacherPromptBuilder` 和 `ProductionPromptBuilder`。
- `teacher` prompt 很长，适合强教师模型；`production` prompt 更短，但还没有明确对应训练/推理。

V4 要求：

- `teacher_eval`：用于教师模型跑分，强调评测任务、答案时机、记忆规则和完整输出格式。
- `teacher_synthesis`：用于教师模型合成 SFT 数据，在 `teacher_eval` 基础上增加“需要保存哪些帧/为什么保存”的引导，以及错误反馈重试说明。错误反馈重试时，要放入上一轮失败输出，并指出结构化错误原因。
- `production`：用于实际训练后模型和 rollout 推理，prompt 简短，只写协议规则和必要行为约束。
- prompt builder 不直接读取全局配置，而是吃 `PromptContext`，这样 eval/synthesis/verl 可以复用。
- SFT 导出时需要能恢复“干净 prompt”。因此 transition 里应区分 `base_prompt`、`retry_feedback`、`attempt_prompt`：失败 attempt 可写 trace，但最终导出合格 SFT 时只用不带错误反馈的 base prompt 还原训练输入。

### 3.6 raw quality、奖励和后处理分离

V3 现状：

- `streamweave/action_quality.py` 会把一部分时序问题记为 warning，例如 note 时间不匹配、事件交叠等；`ValidationReport.valid` 主要由 error 决定。
- `streamweave/action_postprocess.py` 的 `InferencePostprocessor` 会修复 note 时间、bridge 区间、open-tail bridge，并返回实际 memory ops。
- `TeacherPostprocessor` 还没有实现。

V4 要求：

- 对模型原始输出先做 raw quality/reward，不能先修再打分。
- `format_reward` 必须是硬 0/1：必须按固定顺序先输出 `<eta>`，再输出 `<answer>`，之后输出 `note/bridge` 观察标签；只有能抽出 `eta`、`answer`、`note/bridge` 且格式完全符合要求时为 1，任何格式问题都为 0。
- 时序奖励也要显式拆开，例如：
  - `opentail_bridge_reward`
  - `note_bridge_timing_reward`
  - `operation_order_reward`
  - `format_reward`
- note 和 bridge 只要出现交叠、逆序、越界、open-tail 继承错误，就对应奖励为 0。
- 后处理分任务：
  - `synthesis`：不能修改模型输出；如果格式或时序错误，把结构化错误反馈写回 prompt，要求模型重新回答，最多 3 次。3 次 raw output 仍不合格则结束当前样本并标记任务失败。
  - `eval`：可以按规则修复为可执行动作，再进入动作执行和最终评分。
  - `rollout`：可以按规则修复为可执行动作，同时保留 raw reward 供 RL/日志使用。
- eval/rollout 执行动作使用修复后的 `applied_action`，但模型质量统计必须看 raw output 上的 reward。

### 3.7 configs 机制保留，但要分层

V3 的 YAML 配置机制可以保留，但 V4 不应继续把所有字段堆到单个 eval config 里。

建议拆成：

- `runtime`: chunk、fps、resolution、seed、max_steps。
- `backend`: backend 类型、endpoint 列表、并发、timeout、retry。
- `dataset`: dataset 根目录、抽帧 fps、视频源目录、是否预抽帧。
- `prompt`: prompt profile 和模板参数。
- `postprocess`: eval/synthesis/rollout 策略。
- `trace`: 输出路径、是否保存 trace.jsonl、是否保存图片。
- `reward`: 格式奖励、时序奖励、答案奖励等。

敏感字段只允许从环境变量或本地未提交配置读取。

### 3.8 策略要灵活可变

V3 现状：

- `streamweave/policies.py` 已有 `streamweave/full`、`note_only`、`bridge_only`、`no_memory`、`simplestream*` 等入口。
- 但 `gated` 当前等价于 note+bridge 读取，`simplestream*` 只是禁用长期 memory，没有真正实现 recent-N frame/window 策略。

V4 要求至少支持：

- `streamweave`：note + bridge 的完整记忆。
- `keyframe_only`：只读 memory 里的 note 图像，不读 bridge 文本。
- `bridge_only`：只用 bridge 文本，不用 note 图像。
- `recent_frames`：只用 dataset 中已经按指定 fps 抽好的最近 5 张图片，不读长期 memory，不回看原始视频。

这里的策略优先定义为 memory/observation 读取策略。`bridge_only` 的含义是 prompt 里不读 note 图像，只读 memory 中的 bridge 文本；这是可行的。V3 `policies.py` 里 `bridge_only` 同时设置了 `write_notes=False`，V4 不应把 read strategy 和 write policy 绑死，默认仍允许模型按协议输出 note/bridge，是否禁止写 note 由单独的 write policy 控制。

### 3.9 dataset 目录和抽帧缓存

V3 参考格式：

- `raw_data/video_frame/v_QOlSCBRmfWY/000078.jpg`
- 帧文件使用六位数字 jpg 命名。

V4 要求：

- 后续数据统一放在 `dataset/` 下，推荐结构为：

```text
dataset/
  {dataset_name}/
    video/
      {video_id}/
        000000.jpg
        000001.jpg
        ...
```

- runtime 读取数据时先检查目标 fps 的帧是否已经存在；存在则直接使用图片路径。
- 如果还没有抽帧，要先对所有数据集或当前指定数据集执行一次抽帧，生成同一命名格式的帧目录。
- 图片在 memory、trace、SFT 中优先用路径表示，只有构造模型请求时才按需加载 PIL/image bytes。
- 需要一个独立的 `FrameStore` / `DatasetManager`，不要把抽帧逻辑塞进 rollout runner。
- runtime 一律面向已抽帧目录工作。若帧目录不存在，先执行抽帧；一旦抽帧完成，后续策略、trace、SFT、eval/rollout 都只引用抽好的图片路径。

### 3.10 SFT 中间字段和导出接口

V3 现状：

- `data/sft/rollout_sft.py` 复刻了部分 rollout 和 memory 状态逻辑。

V4 要求：

- runtime 每步 transition 默认保存 SFT 需要的中间字段，或至少提供稳定 exporter 接口。
- 建议每步保留：
  - `sample_id`、`video_id`、`step_id`
  - `prompt_text`、`prompt_images`
  - `base_prompt_text`、`attempt_prompt_text`
  - `raw_response`
  - `parsed_action`
  - `quality_report` / `reward_features`
  - `applied_action`
  - `memory_before`、`memory_after`
  - `current_frame_paths`
  - `note_paths`、`bridge_text`
  - `retry_feedback` 和 `attempt_index`
- SFT 合成不应再写独立 runner，而是从同一个 env trace 导出。
- V3 SFT 中间表示的核心字段在 `data/sft/rollout_sft.py`：`memory_before`、`qa_history`、`current_frames`、`raw_teacher_xml`、`target_xml`、`quality.raw`、`quality.target`、`prompt.content`、`prompt.images`、`metadata`。V4 transition 至少要覆盖这些字段，并把失败 attempt 也可追踪地记录下来。
- V3 `data/sft/export_llamafactory.py` 最终只导出 `prompt.content` + `prompt.images` -> `target_xml`。V4 合成模式也应只把合格输出导出成 SFT 样本，失败 attempt 只留在 trace/attempt log。

### 3.11 rollout 指标记录

每一条 rollout 需要记录：

- 每次模型调用耗时和总耗时。
- note 数量、bridge 数量、总 step 数。
- `format_reward`，以及后续拆出的 open-tail/timing/order 等 reward。
- eval/rollout 一次确定性后处理中应用的修复次数 `repair_count` 和修复类型 `repair_types`。
- backend endpoint、重试次数、失败原因。
- 最终答案、最终分数、是否因为格式/后处理/样本错误失败。

这些指标应同时写入 per-video trace 和 batch 级 summary，便于定位模型质量问题和后端稳定性问题。

### 3.12 trace 和 memory 实时写入

V3 现状：

- `streamweave/trace_io.py` 已经支持写 `trace.txt`、`memory.txt`，但也会保存 chunk/note 图片。

V4 要求：

- 输出目录为 `output/{实验名称}/{视频id}/`。
- 每一步完整输入输出实时写入 `trace.txt`，包括 prompt 文本、当前帧路径、memory_before、raw output、quality/reward、postprocess、memory_after。
- 每一步刷新 `memory.txt`，保证中途中断时也能看到最新 memory。
- note 只记录图片路径，不默认保存图片副本。
- 可选保留 `trace.jsonl` 作为机器可读日志，但人工检查优先看 `trace.txt`。

## 4. V4 重构目标

### 4.1 抽出真正的 StreamWeave 环境层

目标新增清晰的环境 API：

```text
StreamWeaveEnv.reset(sample)
StreamWeaveEnv.build_prompt()
StreamWeaveEnv.step(raw_action)
StreamWeaveEnv.snapshot()
```

环境层负责：

- query 到达和 QA history 更新
- memory 状态维护
- 当前 step frame group 管理
- raw action 解析和 raw quality/reward
- 按 profile 调用 postprocess
- transition 记录

模型调用、batch 调度、eval scoring、SFT 导出、RL agent loop 都不应该复制环境逻辑。

### 4.2 统一 schema

V4 应明确区分：

- `raw_action`: 模型原始输出
- `parsed_action`: parser 结果
- `quality_report`: 原始协议质量和结构化错误
- `reward_features`: `format_reward`、`opentail_bridge_reward`、`note_bridge_timing_reward` 等
- `applied_action`: 后处理后实际应用到 memory 的 action
- `memory_ops`: note / bridge / answer 写入操作
- `memory_before` / `memory_after`
- `prompt_text` / `prompt_images` / `current_frame_paths`
- `backend_result`: latency、endpoint、attempt、retry_errors

这套 schema 后续同时服务 eval trace、SFT step 数据、RAFT candidate 采样、RL reward 和 verl agent loop。

### 4.3 后端层独立成可替换模块

建议模块边界：

```text
backends/
  base.py              # Backend, BackendResult, BackendRequest
  openai_compatible.py
  local_http.py
  gemini.py
  pool.py              # BackendPool / endpoint selector / concurrency
  retry.py             # RetryPolicy
```

后端层内部处理图像编码、分辨率、timeout、retry、endpoint 选择和 latency 统计。调用方只拿结构化结果，不关心具体服务类型。

### 4.4 prompt、quality、postprocess 都用 profile

V4 中三者要独立配置：

- `prompt_profile`: `teacher_eval`、`teacher_synthesis`、`production`
- `quality_profile`: 控制哪些格式/时序问题是硬 0 分
- `postprocess_profile`: `synthesis_feedback`、`eval_repair`、`rollout_repair`、`reward_only`

原则：raw quality 永远在 postprocess 前执行；postprocess 不能覆盖 raw reward。

### 4.5 重构 memory 和 policy

V4 需要避免主 runtime 和 SFT 各写一套 memory。

建议把 memory 分为：

- 结构状态：notes、bridges、qa_history
- 渲染策略：streamweave、keyframe_only、bridge_only、recent_frames、no_memory
- eviction 策略：视觉帧 eviction、文本长期保留、窗口裁剪

### 4.6 为 verl no-concat RL 做准备

V4 需要直接支持 `verl修改计划.md` 中的非拼接 rollout：

```text
log pi(traj) = sum_t log pi(action_t | prompt_t)
prompt_t = render(memory_t, qa_history_t, current_frames_t)
memory_{t+1} = env.step(action_t)
```

重点要求：

- 每个 step prompt 独立构造。
- 前一步影响只能通过 memory/qa_history 进入下一步。
- 不把 `prompt_0 + action_0 + prompt_1 + action_1 ...` 拼成训练上下文。
- reward 可以从完整 trace 或 final transition 中读取。

## 5. V4 优先改造顺序

1. 搭建 V4 项目骨架和统一 schema：config、sample、frame、memory、action、transition、reward、trace。
2. 实现 `FrameStore` / `DatasetManager`，先支持读取已抽帧目录和 `resolution=0` 图像路径流。
3. 实现 backend base、OpenAI-compatible backend、local HTTP backend、`BackendPool`、backend 内部 retry。
4. 实现 `StreamWeaveEnv` step API，先用 mock backend 跑通一条最小 trace。
5. 迁移 parser，重写 raw quality/reward，把格式奖励和时序奖励拆成硬 0/1 字段。
6. 实现 postprocess profiles：`synthesis_feedback`、`eval_repair`、`rollout_repair`。
7. 实现三套 prompt profile：`teacher_eval`、`teacher_synthesis`、`production`。
8. 统一 memory rendering 和 policy，补齐 `keyframe_only`、`bridge_only`、`recent_frames`。

10. 重写 SFT step 导出，让它直接消费 transition/trace。

## 6. 修改时的硬边界

- 不再修改 `streamweave_v3`。
- 不把 V3 的运行产物、输出日志、大型数据复制进 V4。
- 不复制明文 API key。
- 每次重构优先保证可运行 smoke，再扩大功能。
- raw quality/reward 必须基于模型原始输出，不能用修复后结果倒推奖励。
- 数据合成模式不能修改模型输出，只能生成错误反馈并请求重答。
- 如果为了复现必须参考 V3，先读 V3，再在 V4 中重新实现或有选择地迁移。

## 7. 我的建议

- 不要从复制 V3 runner 开始。V3 的 `RolloutRunner.run_chunks` 是当前耦合的核心来源，V4 应先建 schema、env、backend、trace 骨架，再迁移可复用函数。
- 后端 retry 和 batch retry 要同时保留但分清语义：backend retry 处理单次请求瞬时失败，batch retry 只处理整条 sample 的灾难性失败。
- `format_reward`、`opentail_bridge_reward`、`note_bridge_timing_reward` 应在最早阶段落成结构化字段，后面 eval、SFT、verl 都直接复用，避免每条管线各自解释格式错误。
- SFT 合成的“错误提示重答”应该由 `quality_report` 自动生成，不要手写字符串判断。这样后续改格式规则时，合成 prompt 的反馈也能同步。
- trace 里优先记录路径，不保存图片副本。V3 `trace_io.py` 有保存 note/chunk 图片的逻辑，V4 默认应关闭，只在 debug 配置打开。
- `recent_frames` 最好作为 observation strategy，而不是 memory policy。它应该控制当前送入模型的帧窗口；memory 是否读写是另一个维度。
- 第一版不要急着做复杂 judge。先把格式、时序、动作执行、trace 和 SFT 导出做稳定，再扩展答案语义评分。

## 8. 代码复查补充风险

这部分来自 2026-04-30 对 V3 代码的再次检查。

### 8.1 strict parser 和 lenient parser 要分开

V3 `streamweave/parser.py` 是宽松正则解析：

- 不检查 `<eta>`、`<answer>`、observation tags 的整体顺序。
- 不检查重复 `<eta>` / `<answer>`。
- 会忽略 `<think>` 和 HTML comment。
- malformed note/bridge 可能只是 parse warning，而不是硬失败。

V4 需要两个入口：

- `strict_validate_raw_output`：只服务 raw reward，严格检查固定顺序、重复标签、XML 外文本、tag 闭合、字段类型，任何问题 `format_reward=0`。
- `parse_for_repair`：服务 eval/rollout 的修复执行，可以宽松抽取可用动作，但不能改变 raw reward。

### 8.2 SFT 合成不能沿用 V3 的 target_xml 逻辑

V3 `data/sft/rollout_sft.py` 会先跑 inference postprocess，再导出 `target_xml = action_to_xml(processed.action)`。这意味着 target 可能是修复后的动作。

V4 合成模式已经明确不能修改模型输出，因此：

- 合成导出目标必须来自 raw output 中已经合格的那次 attempt。
- 失败 attempt 只进 trace/attempt log。
- `target_xml` 可以做等价规范化排版，但不能修时间、修 frame、补 bridge、替换 open-tail。

### 8.3 note 的 frame id 是 step-local，不是全局帧号

V3 prompt 中 Current frames 从 `frame id="1"` 开始重新编号，模型输出 `<note frame="N"/>` 也是当前 step 内编号。SFT 中间表示会另外保存 `global_frame_id` 和 `image_path`。

V4 必须显式保存映射：

```text
step_local_frame_id -> global_frame_index -> image_path -> time_range
```

trace 和 SFT 中 note 应记录 image path；模型输出协议里仍可使用 step-local frame id。

### 8.4 时间计算必须绑定 sample_fps

V3 合成代码多处默认 1fps，并在 prompt 中写过 “Frame id equals timestamp in seconds”。V4 如果保留可配置 fps，需要统一时间公式：

```text
start = (frame_id - frame_id_base) / sample_fps
end = start + 1 / sample_fps
```

在非 1fps 时，prompt 不能再写 frame id 等于秒数。

### 8.5 FrameStore 要检查完整性，而不是只检查目录存在

V4 读取 dataset 时不能只判断 `{video_id}/` 文件夹存在，还要检查：

- 是否有连续帧文件。
- 帧数是否满足 annotation / manifest 要求。
- 文件命名是否符合 `{frame_id:06d}.jpg`。
- `sample_fps`、`frame_id_base`、`frame_count` 是否一致。
- 抽帧中断留下的半成品要能识别并重新抽。

### 8.6 trace 需要先写输入，再写输出

V3 trace 是 step 完成后写入。如果模型调用卡死或崩溃，当前 step 的 prompt 可能没有落盘。

V4 需要分阶段实时写：

```text
step_start: 写 prompt / current_frame_paths / memory_before
backend_done: 写 raw output / backend_result / latency / retry_errors
env_done: 写 reward / postprocess / memory_after
```

每次写入都要 flush，`memory.txt` 每个 step 刷新。

### 8.7 context length 只作为风险观察，不纳入当前需求

V3 note 中记录过 context length error，但用户当前没有要求在 V4 中加入 memory budget 或 prompt budget。V4 第一版不主动实现 bridge 文本、QA history、note 数量或 prompt token 的预算控制。

当前只保留必要的既有行为：按策略渲染 memory，按已抽帧路径构造 observation，按 trace 记录实际输入输出。如果后续评测再次出现上下文超长，再单独讨论是否增加预算控制。

### 8.8 Benchmark sample metadata 不能丢

V3 OVO adapter 中 forward 类任务会把一个 annotation 拆成多个 `{annotation_id}_{test_index}` sample；metadata 里有 `category`、`task`、`ground_truth`、`query_timestamp`、`target_timestamp`。

V4 sample schema 不能只保留 `video_id/question`，必须保留 benchmark metadata，方便 eval、trace、失败过滤和 summary。

### 8.9 多 query / QA history 边界要保留

V3 `BenchmarkSample.query_events` 是 list，runner 会按 chunk 注入 QA history。V4 env 也应支持多 query event，而不是假设每条视频只有一个问题。

需要明确 query 注入规则：

- 当 current step 覆盖 query timestamp 时注入。
- 边界建议使用 `[start, end)`，最后一帧/最后一步可特殊包含。

### 8.10 后端和合成 VLM 代码要统一

V3 主 runtime 用 `streamweave/backends.py`，数据合成又有 `data/synthesize/vlm_client.py`，两边都有图像缩放、OpenAI-compatible、Gemini 逻辑。

V4 应统一到同一套 backend/request/image preprocess 代码，避免 resolution、retry、timeout、endpoint pool 在 eval 和 synthesis 中行为不同。

### 8.11 单元测试需要先补基础模块

V3 `pyproject.toml` 配了 tests，但当前代码库没有测试文件。V4 第一批测试应该覆盖：

- strict format validator
- parser lenient repair extraction
- note frame id 到 path 映射
- open-tail bridge 奖励
- overlap/逆序/越界奖励
- `resolution=0`
- FrameStore 完整性检查
- backend retry 统计
- trace 分阶段实时写

## 9. 已确认决策

- 输出格式固定顺序：`<eta>` 必须在 `<answer>` 前，观察标签在二者之后。
- 抽帧目录可直接由连续编号和 `sample_fps` 推导时间；1fps 时第一张为 `0-1s`，不强制写 `frames.json`。
- 本地 OpenAI-compatible/vLLM 的 8B/8GPU 场景支持 8 个 endpoint；Gemini/API 场景不需要 8 个 endpoint，直接并发同一后端配置。
- SFT 合成重试 prompt 要包含上一轮失败输出和错误原因；失败 attempt 存 trace，只有合格输出进入 SFT 导出。
- eval/rollout 执行修复后的 action；raw reward 只基于模型原始输出统计。
- `bridge_only`、`keyframe_only`、`recent_frames` 首先是 memory/observation 策略，不等同于输出格式限制。
- 空 `<answer></answer>` 合法；但 `<answer>` 标签必须存在，并且必须在 `<eta>` 之后。
- `recent_frames` 的 recent 5 指 dataset 中已经按指定 fps 抽好的图片序列里，当前 step 截止位置之前/包含当前的最近 5 张图片。例如当前处理到 `000016.jpg`，则读取 `000012.jpg` 到 `000016.jpg`。不读取原始视频帧。
- `teacher_synthesis` 不额外保存 selected-frame reason。是否保存帧只体现在模型输出的 note 和 transition 中的 note path。
- SFT 合成的模型重答最多 3 次。3 次 raw output 仍不满足要求时，结束当前样本并标记任务失败，后续由过滤逻辑丢弃。
- eval/rollout 不做 3 次重答，也不做多轮修复；只做一次确定性规则修复，然后执行修复后的 action。需要记录 `repair_count` 和 `repair_types`，这里的 `repair_count` 表示一次后处理中应用了多少条修复规则，不是重试次数。

# 数据清洗 0510

## 文档定位

这份文档用于持续记录 2026-05-10 之后的新数据集扩充、难度测量、清洗过滤和训练数据划分决策。

核心背景：

1. 之前实验发现，现有数据集难度分布非常不均衡，且总量偏少。后续决定扩充数据集；新数据即将处理完成。在新数据正式给出前，代码先假设数据格式与当前 OVO-like / 单视频单问题格式相近。
2. 接下来主线不再使用之前的数据集，而是切到新的扩充数据。
3. 为了摸清新数据集难度，将使用 SFT 后的模型对所有样本统一推理 3 次，并为每条样本增加成功率字段。之后按固定视频/问题时长分布和难度比例分布划分数据，用来分配训练数据。

我的判断：

- 这个方案是合理的。三次采样的成功率可以近似反映样本难度和模型稳定性，比单次推理结果更稳。
- 按时长分布控制数据，可以避免训练集被短视频或长视频单侧主导。
- 按难度比例控制数据，可以避免训练集全是 easy 样本，也避免 hard/unsolved 样本过多导致训练信号太噪。
- 需要保留 `keep_review` 层，不要只做二分类过滤，因为三次采样中 2/3 成功、答案不一致、或格式偶发失败的样本，对后续训练仍可能有价值。

## 目标

对后续大批量新数据做统一闭环筛选，优先筛出适合继续 SFT/RL 的高质量样本，避免把格式不稳、答案不确定、时间标注可疑或任务不可解的数据继续放大。

当前计划：

- 使用已经 SFT 后的 StreamWeave 模型做统一推理。
- 对每条数据采样 3 次。
- 适当调高温度，观察模型在同一输入上的稳定性。
- 对 `dataset2` 的最终合并文件，只在原始 annotation 行上增加 `pass_count`。
- 每次推理的完整 trace 单独落盘，方便抽查轨迹。
- 后续按得分率和错误类型做过滤，而不是只看单次结果。

## 0510 课程式分配策略

当前建议策略：

- `easy`：短样本直接进 SFT，长样本进 RL。
  - 理由：短 easy 样本适合稳定格式和基础回答能力；长 easy 样本虽然答案稳定，但更能训练长程 memory 和 closed-loop 行为，适合 RL。
- `medium` / `hard`：按长度加权抽样，一部分进 SFT，一部分进 RL。
  - 理由：中难题和难题既需要通过 SFT 学到正确轨迹，也需要通过 RL 优化 closed-loop 行为；长度加权可以提高长视频样本进入训练的概率，避免训练集被短样本主导。
- `unsolved`：先不直接进训练，交给更强教师模型推理 3 次。
  - 如果强教师 3 次都不对，直接丢弃。
  - 如果强教师至少 1 次正确，保留正确轨迹用于 SFT，同时该样本也进入 RL。
  - 理由：student 0/3 的样本可能是高难样本，也可能是错标、不可解或视频/时间戳有问题。强教师救回可以区分“有价值的难题”和“脏数据”。

初始 bucket 定义：

```text
easy     = student pass_rate = 3/3
medium   = student pass_rate = 2/3
hard     = student pass_rate = 1/3
unsolved = student pass_rate = 0/3
```

初始分配超参：

```text
easy_sft_max_seconds = 60
medium_sft_ratio = 0.4
medium_rl_ratio = 0.6
hard_sft_ratio = 0.5
hard_rl_ratio = 0.5
length_weight_power = 1.0
min_parser_ok_rate = 2/3
min_quality_valid_rate = 2/3
```

注意：

- `medium/hard` 默认 SFT 和 RL 不重叠；如需允许同一样本同时进入 SFT/RL，可加 `--allow-overlap`。
- `unsolved` 被教师救回后默认同时进入 SFT 和 RL，这是刻意设计，因为它们是 student 当前不会但 teacher 能解的高价值样本。
- 强教师救回时必须保留每次正确 run 的 trace 路径，后续 SFT 需要从正确轨迹导出训练样本。

## 推理协议

模型输出协议仍然是：

```xml
<state>...</state>
<answer>...</answer>
<anchor t="..."></anchor>
<delta t="...">...</delta>
```

硬规则：

- 空 Memory 第一轮必须输出首帧 `<anchor>`。
- `<anchor>` 必须是成对标签，不能使用 `<anchor .../>`。
- `<delta>` 需要覆盖合法时间 gap。
- 最终评测取最后一个非空 `<answer>`。
- 旧内部命名 `note/bridge` 只保留在代码 schema/指标里，不作为模型可见协议。

初始建议：

- `pass_rate = pass_count / 3`。
- `parser_ok_rate` 用来判断模型是否能稳定遵守 XML 协议。
- `quality_valid_rate` 用来判断 anchor/delta 时间和初始化规则是否稳定。
- `answer_consistency` 记录 3 次最终答案是否一致。

## 初始过滤策略

先按保守规则分层，不急着一次性定死：

```text
keep_strong: pass_rate = 1.0 且 parser_ok_rate = 1.0 且 quality_valid_rate = 1.0
keep_review: pass_rate >= 2/3，但存在格式、时间或答案不一致问题
drop_weak: pass_rate <= 1/3
drop_invalid: parser_ok_rate = 0 或 quality_valid_rate = 0
```

需要人工复核的情况：

- 三次答案不一致，但至少两次得分正确。
- 格式修复后能跑通，但 raw output 经常不合法。
- 问题时间点、视频路径或帧目录可疑。
- GT 本身疑似错标或选项格式不规范。

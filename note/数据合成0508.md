# 数据合成 0508

日期：2026-05-08

## 目标

从数据源开始梳理 OVO-Bench 和 StreamingBench 的上游数据来源，为后续 StreamWeave 新数据合成做记录。

## 结论先行

OVO-Bench 是“多上游数据集 + 自抓 YouTube”改造成 online video understanding 的评测集。它的核心不是简单 QA，而是把已有视频、QA、动作片段、事件时间戳改造成带 `query_time`、`clue_time`、`answer_time` 或 `evidence_interval` 的在线样本。

StreamingBench 不是同类的多数据集合并。它主要是人工挑选约 900 个 YouTube 视频，再用 GPT-4o caption/QA 生成加人工校验，最终形成 18 类任务、4500 个 QA。它更像“从公开视频重新标注”，不是从已有 QA benchmark 直接转换。

对我们来说，OVO-Bench 的上游更适合直接复用成 StreamWeave 合成源，因为很多来源已经有事件时间戳或可恢复时间结构；StreamingBench 更适合参考任务 taxonomy 和 QA 生成流程。

## OVO-Bench 上游来源

论文/补充材料给出的来源分两类。

第一类是已有人工标注数据集，只取 val/test split，降低和训练数据重合的风险：

| OVO 任务 | 上游数据集 |
| --- | --- |
| `EPM` Episodic Memory | `QA-Ego4D`, `OpenEQA` |
| `ASI` Action Sequence Identification | `STAR`, `YouCook2`, `CrossTask`, `HiREST`, `COIN` |
| `REC` Repetition Event Count | `Perception-Test`, `THUMOS14/15` |
| `SSR` Sequential Steps Recognition | `COIN` |
| `CRR` Clues Reveal Responding | `MovieNet` |
| Real-Time Visual Perception: `STU/OJR/ATR/ACR/OCR/FPD` | `Ego4D` |

第二类是 self-crawled YouTube videos，用来补充 domain diversity。补充材料 license 表还列了 `E.T.Bench`，但当前本地 `ovo_bench_new.json` 没看到对应视频前缀，先不把它当作主来源。

OVO 的 meta-annotation 构造路径：

1. 直接复用已有 event-level timestamp：`QA-Ego4D`, `COIN`, `Ego4D`。
2. 半自动补时间戳：`OpenEQA`, `STAR`, `Perception-Test`, `THUMOS` 等先用 Gemini-1.5 生成粗时间戳，再人工修正。
3. 纯人工标注：`SSR`, `CRR` 的问题、答案和 ground-truth timestamp 主要人工构造。

## StreamingBench 上游来源

StreamingBench 的上游主要是人工筛选的 YouTube 视频，不是从多个已有 QA benchmark 合并。论文的数据构造流程：

1. 手工选择并整理约 `900` 个 YouTube 视频。
2. 覆盖 8 类场景：life record, competition, education, TV show, video games, documentary, animation/movie, unusual event。
3. Real-Time Visual Understanding 和 Proactive Output：1 fps 抽帧，给帧加时间戳，用 GPT-4o 生成细粒度 captions，再生成 QA 和 question timestamp。
4. Omni-Source Understanding、SQA 和部分 Contextual Understanding：人工标注 QA。
5. 所有 QA 经过人工质检，删除不需要视频就能回答、歧义或错误的问题。

任务结构：

- Real-Time Visual Understanding：10 类任务，官方本地 `questions_real.json` 有 `2500` 个问题。
- Omni-Source Understanding：6 类任务，官方本地 `questions_omni.json` 有 `1500` 个问题。
- Sequential Question Answering：官方本地 `questions_sqa.json` 约 `250` 个 SQA 问题，按 50 条 sequence 组织。
- Proactive Output：官方本地 `questions_proactive.json` 有 `250` 个问题；`questions_proactive_50.json` 是评测子集。

当前 StreamWeave v5 adapter 状态：

- `exp3/streamweave_v5/evaluation/streamingbench_adapter.py` 现在只注册了 `real`, `sqa`, `proactive` 三个 split。
- 官方 `questions_omni.json` 暂时没有被 adapter 接入。
- 如果后续要把 StreamingBench 全量纳入 StreamWeave eval 或数据合成，需要补 `omni` split，并考虑 audio 相关字段；否则只能复用视觉/上下文部分。

## 对 StreamWeave 数据合成的启发

优先做 OVO-style 合成，因为它和 StreamWeave 的在线 memory/streaming 逻辑更贴近。关键中间 schema 不应该只有最终 QA，至少要保留：

- `source_dataset`
- `source_video_id`
- `video_path`
- `task_type`
- `query_time`
- `evidence_time` 或 `evidence_interval`
- `answer_time`，只在 forward active responding 需要
- `question`
- `answer`
- `options`
- `negative_options_source` 或 `misleading_time`
- `quality_check_status`

推荐第一批优先看这些来源：

1. `COIN` / `YouCook2` / `CrossTask` / `HiREST`：适合 procedure、step recognition、sequence memory。
2. `Ego4D` / `QA-Ego4D` / `OpenEQA`：适合 egocentric episodic memory、query after observation、hallucination detection。
3. `THUMOS14/15` / `Perception-Test`：适合 repetition/action/event count。
4. `MovieNet`：适合 clue reveal / delayed answer，但时间跨度长、人工成本更高。
5. 自抓 YouTube：适合补 domain diversity，但必须单独建立版权、下载、去重、质量筛查记录。

## 需要继续确认的问题

1. 我们本机是否已经有上述上游源数据的原始视频和标注，而不只是 OVO/StreamingBench 的成品标注。
2. 新数据是直接转换已有 QA，还是重新从视频合成 StreamWeave-style trajectory。
3. 是否需要同步支持 StreamingBench omni-source，即音频相关问题；如果支持，当前 StreamWeave 主链路和 RL 链路都要补 audio 输入逻辑。
4. 对每个合成样本要不要强制保存 human/LLM 质检日志，方便后续 RL 出错时反查源数据。

## 0508 追加：流式视频理解数据合成调研

这部分记录现有 streaming/online video understanding 工作如何构造数据，以及哪些开源工作值得借鉴。

### 总体模式

现有工作大致分五类：

1. 复用已有带时间结构的数据集，转换成流式 QA 或流式对话。
2. 从公开视频或已有视频库筛选长视频，再用 GPT-4o/Gemini 生成 timestamped QA，最后人工校验。
3. 把长视频切成 scene/event segments，在每个 segment 生成多轮 QA chain，再人工建立跨 segment temporal link。
4. 用 ASR/closed captions 做便宜的大规模流式预训练，按时间戳把 speech tokens 和 video frames interleave。
5. 专门构造 forward/proactive 样本，训练或评测“什么时候回答/什么时候保持沉默”。

对 StreamWeave 最有价值的是第 1、3、5 类：它们能自然对应我们的 interleaved memory、stepwise rollout、format/answer reward 和 proactive/online response gate。

### 可直接借鉴的开源工作

#### VideoLLM-online / LIVE

链接：

- GitHub: https://github.com/showlab/videollm-online
- Dataset: https://huggingface.co/datasets/chenjoya/videollm-online-chat-ego4d-134k

数据合成方式：

- 核心思路是把 offline temporal annotation 转成 streaming dialogue。
- 使用 Ego4D Narration Stream `113K`，并用 Llama-3 清洗/改写 narration，去掉 Ego4D 原始 narration 里的特殊符号。
- 使用 Ego4D GoalStep annotation 生成 `21K` streaming free-form dialogue。
- 训练数据不是普通“看完整视频再回答”，而是沿时间流逐步输入 frame/text，让模型在流中持续语言建模。

可借鉴点：

- 可以看 `data/livechat/` 的生成脚本，把我们已有的 OVO/COIN/YouCook2/HiREST 时间戳样本改造成 StreamWeave 多轮轨迹。
- 它的优势是便宜、可扩展；不需要每条都用昂贵闭源 VLM 重新看视频。
- 对我们最像的是“把已有事件时间戳变成流式对话/轨迹”，不是“重新发明一个 benchmark”。

风险：

- 它更多训练 continuous narration / free-form dialogue，不一定直接对应我们的 `<eta><answer>` 决策格式。
- 需要把它的 streaming dialogue 进一步规整成 StreamWeave 的 query event、memory update、answer/abstain 标签。

#### TimeChat-Online-139K

链接：

- Project: https://timechat-online.github.io/
- Dataset: https://huggingface.co/datasets/yaolily/TimeChat-Online-139K

数据合成方式：

1. 从 13 个公开视频数据集收集 `11,043` 个 visually informative videos。
2. 抽 scene-oriented key frames，平均每个视频 `87.8` 个关键帧，约每 `7.14s` 一个。
3. 用 GPT-4o 为关键帧生成 dense captions，平均每个 key frame caption 约 `176` words。
4. 基于 dense captions 合成 streaming VideoQA。
5. 为 future-response training 构造 negative samples。

数据来源：

- `COIN`, `QV-Highlights`, `ActivityNet`, `HD-VILA`, `YouCook2`, `TVSum`, `ViTT`, `QuerYD`, `YouMakeup`, `VideoIC`, `Movie101`, `HiREST` 等。

可借鉴点：

- 这是最接近我们目标的开源数据合成范式：长视频、多来源、三类任务：backward tracing、real-time visual perception、forward active responding。
- 它的“关键帧 dense caption -> QA”可以直接变成我们第一阶段合成 pipeline。
- 它专门构造 future-response negative samples，这对我们解决“过早回答/应该等待”很重要。

风险：

- 只发布 1fps extracted frames，不一定有完整原视频。
- 机器生成比例高，需要我们加自己的 quality gate，否则 RL 会学到 caption artifact。

#### SVBench

链接：

- GitHub: https://github.com/sotayang/SVBench
- Paper: https://ar5iv.labs.arxiv.org/html/2502.10810

数据合成方式：

1. 从 `YT-Temporal-1B`, `YouCook2`, `ActivityNet`, `MovieChat`, `Panda-70M`, `Ego4D` 等来源收集 `12,989` 个 raw videos。
2. 用 aesthetic score、optical flow score 过滤视频。
3. 用 scene detection 把视频切成 scenes，保留 5-15 scenes 的视频，最终 `1,353` 个视频。
4. 每个 clip 用 GPT-4o 生成 5-6 轮 QA chain。
5. 人工增删改 QA，保证多轮问题之间有上下文连接、代词和实体引用合理。
6. 用 GPT-4 从 accuracy、completeness、relevance、fluency、contextual comprehension、logical consistency、temporal understanding 七个维度打分；低于阈值的 chain 返修。
7. 进一步识别相邻 QA chains 的 temporal linkages，再人工修改形成跨 clip temporal dialogue paths。

可借鉴点：

- 它开源了 `construct_data/`，很适合借鉴“生成 QA chain + temporal linkage”的流程。
- 这个方向对 StreamWeave 的 memory 很有用：模型必须引用历史 QA/历史片段，而不是只看当前帧。
- 适合做多轮 SFT 和评测，不一定适合直接做短 horizon RL。

风险：

- 人工环节很重。
- 它的问题更偏多轮 dialogue/path reasoning，不是严格 OVO-style online query timing。

#### CogStream

链接：

- GitHub: https://github.com/LiamZhao326/CogStream
- Dataset: https://huggingface.co/datasets/SII-KYW/CogStream
- Paper: https://ojs.aaai.org/index.php/AAAI/article/view/38336

数据合成方式：

- 来源：`MovieChat`, `MECD`, `QVhighlights`, `VideoMME`, `COIN`, `YouCook2`。
- 最终 `1,088` videos / `59,032` QA pairs。
- 先人工筛选有清晰故事线、因果关系、多步发展的长视频。
- 用 SceneTiling 按 semantic boundary 切成 event segments。
- 半自动生成 Basic / Streaming / Global 三类 QA。
- 大部分 QA 用 GPT-4o 或 Gemini-2.0-flash 生成，再人工复核。
- Dynamic Updating 类问题完全人工设计和修正，因为它要求答案随视频流动态变化。
- 每个问题显式标出 relevant historical QA/context，用于训练/评测“检索相关历史而不是塞入全部历史”。

可借鉴点：

- 非常适合我们做 memory 可观测性和 memory retrieval 训练：它的关键是“哪些历史是相关的”。
- 可以借鉴它的 Basic/Streaming/Global 分层标签，给 StreamWeave 样本加 difficulty/context_dependency 标签。

风险：

- 任务核心是 context-guided reasoning，不一定强调 forward active response 的时机。
- 如果照搬，会偏向历史对话检索，不一定覆盖实时视觉 perception。

#### VStream-QA / Flash-VStream

链接：

- Paper: https://arxiv.org/abs/2406.08085
- Dataset: https://huggingface.co/datasets/IVGSZ/VStream-QA
- GitHub: https://github.com/IVGSZ/Flash-VStream

数据合成方式：

- 基于 `Ego4D` 和 `MovieNet`。
- 分 offline `VS-Ego/VS-Movie` 和 online `RVS-Ego/RVS-Movie`。
- online annotation 每条包含 `start_time`, `end_time`, `gt_duration`, `duration`。
- 规模：`VS-Ego` 246 videos / 1465 Q，`VS-Movie` 83 videos / 1905 Q，`RVS-Ego` 99 videos / 1465 Q，`RVS-Movie` 1000 videos / 1905 Q。

可借鉴点：

- 简洁：它证明最小的 online QA schema 至少要有 evidence interval。
- 可以作为我们 schema 设计的 lower bound：`video_id/question/answer/answer_type/start_time/end_time/duration`。

风险：

- 任务类型相对少，主要是 QA，不够覆盖 StreamWeave 的动作修复、memory update、forward response。

#### StreamingBench

链接：

- GitHub: https://github.com/THUNLP-MT/StreamingBench
- Paper: https://arxiv.org/abs/2411.03628

数据合成方式：

- 人工选择 `900` 个 YouTube 视频。
- 每个视频 5 个问题，共 `4500` QA。
- Real-Time Visual Understanding 和 Proactive Output：1fps 抽帧，加时间戳，用 GPT-4o 先生成每 20 帧 captions，再生成 QA 和 question timestamp。
- Omni-Source Understanding、SQA 和部分 Contextual Understanding：人工标注。
- 所有 QA 人工质检，删掉不用视频即可回答、歧义或错误问题。

可借鉴点：

- 任务 taxonomy 完整，适合参考 real-time visual、audio-visual、contextual/proactive 分类。
- 对我们可以借鉴 timestamped frame caption -> QA 的流程。

风险：

- 上游视频主要来自 YouTube，源列表和版权管理要单独处理。
- Omni-source 涉及 audio，当前 StreamWeave 主链路/RL 链路还没有完整 audio 输入。

#### OmniMMI

链接：

- Project: https://omnimmi.github.io/
- Paper: https://arxiv.org/abs/2503.22952
- Dataset: https://huggingface.co/datasets/bigai-nlco/OmniMMI

数据合成方式：

- 来源包括 `Ego4D`, `COIN`, `Shot2Story20K`, `QVHighlight`, `MLVU`，外加用 425 个关键词从 YouTube 补充互动场景视频。
- 最终 `1,121` videos / `2,290` questions，平均视频超过 5 分钟。
- 人工按时间顺序观看视频，标注相关 action/state time span。
- 二次人工 review 检查 question、answer、time span 一致性。
- 任务分 streaming video understanding 和 proactive reasoning，包括 SG/AP/MD/SI/PA/PT。

可借鉴点：

- 它强调“交互”：multi-turn dependency、turn-taking、proactive alerting、speaker identification。
- 如果我们后面要做语音/音频/主动开口，这个 benchmark 的任务定义值得参考。

风险：

- 多模态 audio/speech 强绑定，当前 StreamWeave 暂时是视觉流式理解，直接搬会引入新链路复杂度。

#### PhoStream

链接：

- Dataset: https://huggingface.co/datasets/lucky-lance/PhoStream
- GitHub: https://github.com/Lucky-Lance/PhoStream

数据合成方式：

- mobile-centric streaming benchmark，`578` videos / `5572` open-ended QA。
- 覆盖 on-screen/off-screen 手机场景，强调 video、audio、temporal reasoning。
- 用 Automated Generative Pipeline 生成 timestamped QA，再用自动 checker 和两轮人工 verification。
- 任务里显式区分 Instant、Backward、Forward，Forward 强调不能提前回答。

可借鉴点：

- 它直接把 Early Response 作为核心失败模式。这个和我们 RL 里的 format gate / answer timing 很相关。
- 如果做 forward-active 数据，应该像它一样记录 early response / no response / correct timing 三类评估指标。

风险：

- 很新，代码/数据完整度需要本地实际拉取后确认。
- 依赖 audio/phone scenario，未必适合第一版视觉-only StreamWeave。

### 推荐我们优先借鉴的路线

第一优先级：TimeChat-Online + VideoLLM-online。

- 用 TimeChat-Online 的“多来源视频 -> scene keyframes -> GPT-4o dense caption -> streaming QA -> future negative”的大框架。
- 用 VideoLLM-online 的“已有 temporal annotation -> streaming dialogue”的低成本转换方式。
- 这两者最适合快速产出可训练数据。

第二优先级：SVBench + CogStream。

- 用 SVBench 的 QA chain 和 temporal linkage 思路做多轮记忆样本。
- 用 CogStream 的 relevant historical QA/context label 做 memory retrieval/可解释监督。
- 这两者适合提升 StreamWeave memory，而不是第一批全量数据生产。

第三优先级：StreamingBench / OmniMMI / PhoStream。

- 参考任务设计和评价指标。
- 暂时不直接照搬 audio/omni 任务，除非我们决定扩展音频链路。

### StreamWeave 可执行数据合成草案

第一版建议只做视觉流式数据，不引入音频：

1. 选源数据：`COIN`, `YouCook2`, `HiREST`, `Ego4D/OpenEQA`, `QV-Highlights`, `ActivityNet`。
2. 抽帧：统一 1fps，同时保存原视频时长、fps、frame timestamp。
3. 切分：用 scene detection 或已有 step/event annotations 得到 event intervals。
4. 生成中间 caption：
   - 对每个 event/keyframe 生成 dense caption。
   - caption 必须带 timestamp、visible objects/actions/state changes。
5. 生成 QA：
   - Backward：问题在 query_time 问，证据在历史 evidence_time。
   - Real-time：问题依赖当前或最近 window。
   - Forward：query_time 时不能回答，answer_time 后才可回答，同时生成 before-answer negative。
6. 构造 StreamWeave trajectory：
   - 每条样本输出 query events、ground truth answer、evidence intervals、should_answer_by_step、should_wait_steps。
   - 对 forward 样本显式标注 early-answer penalty 区间。
7. 质检：
   - 程序检查时间戳合法、answer_time 不超过视频长度、evidence 在 query 之前或之后符合任务定义。
   - LLM 检查问题是否能被 evidence 支撑、错误选项是否视觉相关。
   - 抽样人工复核。
8. 训练拆分：
   - SFT：使用完整 trajectory + repaired format。
   - RL：使用同 schema，但 reward 分 format、answer correctness、wait/early timing。

### 暂定不做的事

- 暂时不做 audio/omni-source，因为会改变主推理链路和 RL 输入结构。
- 暂时不做纯 YouTube 大规模抓取，先用已公开数据集和本地已有视频源。
- 暂时不直接用机器生成 QA 做 RL reward gold，先通过过滤和抽检得到小而干净的种子集。

## 0508 追加 2：补全流式视频研究谱系

这次补充的重点不是把所有 video benchmark 都列进来，而是找和 StreamWeave 真正相关的三类工作：

1. 数据/benchmark 能直接借鉴：有 streaming timestamp、reply timing、evidence window、interleaved video-text 格式。
2. 训练范式能借鉴：把离线视频标注转成逐帧/逐时间片 supervision。
3. 系统设计能借鉴：memory、trigger、silence/respond、KV cache 等机制，但不一定能直接拿数据。

### 领域核心问题拆分

流式视频理解不是单纯“长视频 QA”。它至少包含两个正交问题：

- `When to act`：什么时候回答、什么时候等待、什么时候主动提醒。典型输出是 `Silence / Standby / Response`、EOS、NO REPLY、readiness score、activation score。
- `How to sustain`：视频一直进来，如何不爆 context/KV cache。典型方法是 sliding window、hierarchical memory、event memory、KV cache retrieval、token pruning、caption/memory summarization。

对 StreamWeave 来说，第一类更贴近 RL reward 和 trajectory 设计；第二类更贴近 memory 结构和推理效率。

### 新增重点工作

#### Streamo / Streaming Video Instruction Tuning

链接：

- GitHub: https://github.com/maifoundations/Streamo
- Paper page: https://huggingface.co/papers/2512.21334
- Dataset: https://huggingface.co/datasets/maifoundations/Streamo-Instruct-465K

数据/训练格式：

- 它公开了 `Streamo-Instruct-465K`，定位是 streaming video instruction-following。
- 原始样本包含 `video_name`, `video_path`, `task_type`, `source`, `question`, `response`。
- `question.time` 表示问题出现时间。
- `response.st_time/end_time` 表示事件开始和结束；`response.time` 用于 instant response。
- 训练格式是 1fps 的多轮 conversation，每一帧一个 user turn，assistant 输出三类状态：
  - `</Silence>`：当前没有相关事件或输入不相关。
  - `</Standby>`：事件正在发生，但还没到回答时机。
  - `</Response>`：事件完成，可以开始回答。

可借鉴点：

- 这是目前最贴近 StreamWeave RL 的格式之一。我们的 `<eta></eta><answer></answer>` 可以扩展成显式状态监督：`silent / standby / answer`。
- 它把“什么时候回答”和“回答什么”放进同一个 autoregressive 序列里，适合做 SFT seed，再接 RL。
- 原始 schema 里已经有 `question.time`, `response.st_time`, `response.end_time`，可以直接映射到我们的 `query_time`, `evidence_time`, `answer_time`。

风险：

- 它的目标是通用 streaming assistant，不一定完全等价 OVO 的 online QA。
- 模型权重未公开，数据和训练代码可借鉴，不能指望直接复现实验。

#### VideoChat-Online / OVBench

链接：

- Project: https://videochat-online.github.io/
- Paper: https://arxiv.org/abs/2501.00584

核心思路：

- OVBench 是 online video understanding benchmark，包含 10 类任务、200 个视频片段、860 个 QA。
- VideoChat-Online 使用 streaming video buffer、hierarchical memory 和实时 ASR，强调在线场景不能看未来帧。
- 任务覆盖 past memory、future prediction、spatial perception，也包含 audio-video 任务。

可借鉴点：

- 它的层级 memory 思路可以对应 StreamWeave 的 interleaved memory：近期细粒度帧保留，远期内容压缩成更抽象的 memory。
- OVBench 任务设计可以作为 OVO/StreamingBench 之外的补充评测来源。

风险：

- benchmark 规模不大，更适合评测和任务定义，不适合作为主要训练数据源。

#### LiveCC

链接：

- Project: https://showlab.github.io/livecc/
- GitHub: https://github.com/showlab/livecc

数据合成方式：

风险：

- 它更偏 video commentary 和 ASR 对齐，不是纯视觉 QA。

#### MMDuet / MMDuetIT

链接：

- GitHub: https://github.com/yellow-binary-tree/MMDuet
- Dataset: https://huggingface.co/datasets/wangyueqian/MMDuetIT
- Paper: https://huggingface.co/papers/2411.17991

数据合成方式：

- 提出 video-text duet interaction：视频连续播放，用户和模型都可以在任意帧后插入文本。

可借鉴点：

- 它的 duet format 和我们流式 prompt 的基本假设一致：视频不是一次性输入，而是和文本交替出现。

风险：

- 它并不是严格的 RL trajectory 框架。
- 主要目标是增强 time-sensitive task，不直接等于 OVO-style online decision。

#### ProAssist

链接：

- Project: https://pro-assist.github.io/
- Paper: https://huggingface.co/papers/2506.05904

数据合成方式：

- 从 annotated egocentric videos 合成 proactive assistant-user dialogues。

可借鉴点：

- 这是“主动指导/什么时候说”的强相关工作。

#### ProactiveVideoQA / ProactiveBench

链接：

- Dataset: https://huggingface.co/datasets/wangyueqian/ProactiveVideoQA
- Paper: https://huggingface.co/papers/2507.09313

数据格式：

- 样本字段包括 `video`, `question_id`, `conversation`, `answer`, `duration`。
- `conversation` 里 user question 有 `time`。
- `answer` 是 list，每个 answer 有：
  - `content`
  - `related_timespan`
  - `reply_timespan`
  - `type`，例如 `step`, `substep`, `substep-merged`。

可借鉴点：

- 这个格式非常适合 StreamWeave：`related_timespan` 是证据窗口，`reply_timespan` 是允许回答窗口。
- 它天然支持多个答案片段，对应 OVO 里一个问题可能有多个正确出现时间。
- 可以直接用来设计我们的 reward：
  - 在 `reply_timespan` 前回答：early penalty。
  - 在 `reply_timespan` 内回答且内容正确：positive。
  - 超过窗口不答：late/no-response penalty。

风险：

- 很多视频是 360 秒切片，和我们当前 lt120s 训练集长度不同。
- 需要确认原视频可获得性。

#### StreamReady / ProReady-QA

链接：

- Microsoft Research: https://www.microsoft.com/en-us/research/publication/streamready-learning-what-to-answer-and-when-in-long-streaming-videos/

核心思路：

- 提出 Answer Readiness Score，强调答案不仅要正确，还要在 supporting visual evidence 出现时回答。
- 早答是 speculation，晚答降低实时性，所以指标对 early/late 使用非对称惩罚。
- ProReady-QA 标注 answer evidence windows 和 proactive multi-turn questions。

可借鉴点：

- 这是我们 RL reward 设计最应该参考的方向之一。
- 我们现在先做 `format_avg + success` 是合理的起步，但后续 answer timing reward 应该接近 ARS：正确性和时机分开算，再组合。

风险：

- 目前看到的是研究页面，数据是否完全公开还要继续确认。

#### LiveStar / OmniStar-RNG

链接：

- GitHub: https://github.com/sotayang/LiveStar
- Dataset: https://huggingface.co/datasets/yzy666/OmniStar-RNG

数据/系统方式：

- LiveStar 关注 always-on proactive responses。
- OmniStar-RNG 提供 10,137 个视频文件，仓库说明里有视频/帧提取流程。

可借鉴点：

#### Dispider

链接：

- GitHub: https://github.com/Mark12Ding/Dispider
- Paper: https://huggingface.co/papers/2501.03218

核心思路：

- 把在线系统拆成 Perception、Decision、Reaction 三个异步模块。
- Perception 持续监控视频；Decision 判断什么时候触发；Reaction 负责生成详细响应。
- 这样避免 autoregressive decoding 阻塞视频感知。

可借鉴点：

- 对 StreamWeave 推理链路有启发：RL 训练时可以先统一成一步一个 action；真实系统后续可以拆成轻量 trigger + 重模型 answer。
- 这也解释了为什么“每 step 都让大模型完整生成”会慢：Decision 和 Reaction 没有解耦。

风险：

- 这是系统架构，不是数据合成 pipeline。
- 训练代码释放状态有限，不能作为主要数据来源。

#### StreamBridge / Stream-IT

链接：

- Paper: https://arxiv.org/abs/2505.05467

核心思路：

- 把 offline Video-LLM 转成 proactive streaming assistant。
- 用 memory buffer + round-decayed compression 管理多轮长上下文。
- 用轻量 activation model 决定什么时候主动响应。
- 构造 Stream-IT，一个 interleaved video-text streaming instruction 数据集。

可借鉴点：

- 对我们最有用的是“activation model”和“interleaved video-text instruction”的组合。
- 如果 StreamWeave 后续要加轻量 gate，可以借鉴它把 `when to speak` 从主模型中解耦出来。

风险：

- 目前没有看到完整可下载数据入口。
- 更适合系统设计，不适合第一版数据合成落地。

#### Think While Watching

链接：

- GitHub: https://github.com/wl666hhh/Think_While_Watching
- Paper: https://arxiv.org/abs/2603.11896

核心思路：

- 增量读取视频片段和问题，边看边生成 thoughts 和 answers。
- 数据格式包含 `video`, `conversations`, `metadata.timestamps`, `segment_info`, `frame_files`, `sample_timestamps`。

可借鉴点：

- 它的 sample schema 对我们调试数据流有参考价值：timestamp、segment、frame file 都显式放在 metadata 里。
- 对 memory 可观测性有帮助，可以记录每个 turn 当前使用了哪些 segment/memory。

#### SimpleStream

链接：

- Project: https://simple-stream.github.io/

核心结论：

- 只用最近 N 帧的 sliding-window baseline，就能在 OVO-Bench 和 StreamingBench 上达到很强结果。
- 它质疑复杂 memory/retrieval/compression 模块是否真正带来收益。

可借鉴点：

- 对我们很重要：StreamWeave 必须和 simple recent-window baseline 做公平对比。

#### AURA

链接：

- Project: https://aurateam2026.github.io/

核心思路：

- Always-on streaming visual interaction framework。
- 使用 dual sliding window：视频窗口保留最近 N 秒帧，QA 窗口保留最近 M 组问答。
- chunk-wise conversational format：每个时间步模型要么输出响应，要么输出 `<|silent|>`。

可借鉴点：

- 和我们当前框架很接近：视频流窗口 + 历史 QA/memory 窗口分开管理。

- 主要是系统/模型框架，是否有完整数据生产代码需要继续确认。

### 其他应记录但暂不展开的相关工作

这些工作和 streaming/online video 有关，但目前对“数据合成第一版”帮助不如上面几项直接：

### 对 StreamWeave 数据合成的修正建议

之前的第一版 pipeline 还不够明确“等待/回答”的监督。现在建议改成下面这个 schema：

```json
{
  "sample_id": "...",
  "dataset": "ovo|streamingbench|synthetic",
  "video_path": "...",
  "duration": 118.3,
  "fps": 1.0,
  "frames": [
    {"idx": 0, "time": 0.0, "path": "..."}
  ],
  "query": {
    "content": "...",
    "time": 12.0
  },
  "answers": [
    {
      "content": "...",
      "evidence_timespan": [42.0, 48.0],
      "reply_timespan": [48.0, 52.0],
      "type": "instant|backward|forward|proactive"
    }
  ],
  "turn_labels": [
    {"step": 0, "time": 5.0, "state": "silence"},
    {"step": 8, "time": 45.0, "state": "standby"},
    {"step": 9, "time": 50.0, "state": "response"}
  ],
  "quality": {
    "source_caption": "...",
    "generator": "...",
    "checker": "...",
    "human_checked": false
  }
}
```

关键变化：

- 不只存 `answer`，必须存 `evidence_timespan` 和 `reply_timespan`。
- 每个 step 最好有 `state`：`silence / standby / response`。这比只有最终 answer 更适合 RL。
- `forward` 样本必须有 early-answer negative region，否则模型不知道什么时候该等。
- SFT 可以学习 `state + answer`；RL 可以先只用 format/success，后续再加 timing reward。

第一版不要直接做的：

- 不要先做复杂 KV cache/retrieval 数据训练。
- 不要把所有长视频 QA 当 streaming QA；必须有 query time、evidence window、reply window。

第一版合成目标要非常明确：做视觉-only、可自动评分、可进入 StreamWeave SFT/RL 的流式样本。不要一开始就混 audio、open-ended judge、复杂主动助手对话、独立 trigger 模块。

```text

-> 抽帧或 scene keyframes
-> dense temporal captions
-> 生成 backward / realtime / forward QA
-> 自动检查 evidence window 和 early-answer window
-> 转成 StreamWeave 统一 schema
```

### 第一版样本类型

`backward`：

- 问题在 `query_timestamp` 出现。
- 答案证据在过去。
- 模型应该立即回答。

`realtime`：

- 答案证据在当前或最近窗口。
- 主要测实时视觉感知。
- 这类样本要避免只靠语言常识能答。

`forward`：

- 问题先出现。
- 答案证据在未来。
- `query_timestamp < evidence_start <= target_timestamp`。
- 必须保留 `negative_before`，否则以后没法判断 early response。

`multi_answer`：

- 一个问题可能有多个正确回答时刻。
- 每个答案必须有自己的 `evidence_timespan` 和 `reply_timespan`。
- 先作为 P1，不放第一批 RL 主训练也可以。

### 建议统一 schema

    {

  "target_timestamp": 35.0,

  "reply_timespan": [35.0, 38.0],

  "question": "...",
  "options": ["...", "...", "..."],
  "ground_truth": "A",
  "answer": "...",
  "answer_events": [
    {
      "content": "...",

### 必须做的自动检查

1. 时间合法性：
   - `0 <= query_timestamp <= duration`
   - `0 <= evidence_start <= evidence_end <= duration`
   - `reply_timespan` 在视频范围内

2. forward 合法性：
   - `query_timestamp < evidence_start`
   - `negative_before = [query_timestamp, evidence_start)`
   - `negative_before` 里不能已经泄露答案

3. 多选唯一性：
   - 正确选项只能有一个。
   - 如果两个选项都能被 evidence 支持，丢掉。

4. 证据支持：
   - answer 必须能从 `evidence_timespan` 的 dense caption 或原视频帧支持。
   - 不能只靠常识生成。

5. StreamWeave 可运行性：
   - query 必须能落到 rollout 会走到的 step。
   - 如果视频被截断，query/evidence/reply 不能落在截断外。
   - 图片帧路径必须存在并可读。

### 最终优先级

P0 先做：

- OVO schema 对齐。
- 多选 QA。
- backward / realtime / forward 三类。
- `evidence_timespan`、`reply_timespan`、`negative_before`。
- 自动 checker。

P1 再做：

- multi-answer。
- `frames_required`。
- `temporal_clue_type`。
- `required_memory_ids`。
- memory hit/miss debug。

P2 暂缓：

- audio。
- open-ended LLM judge。
- timing reward。
- CoT 监督。
- 独立 activation model。
- 复杂异步 perception/decision/reaction 架构。

### 当前硬约束

1. `OVO-Bench` 和 `StreamingBench` 是重点 benchmark，不能直接从它们的成品样本里抽训练数据。
2. 训练/合成数据要尽量和这两个 benchmark 同分布：
   - 对 OVO：优先选择它真实上游重叠的数据集。
   - 对 StreamingBench：它主要是人工筛选 YouTube 视频，没有固定公开上游，因此只能按视频类别和任务类型对齐。
3. 所有候选源必须做 video-level 去重：
   - 原始 video id / YouTube id 去重。
   - 文件名和 source id 去重。
   - 必要时做 perceptual hash 或 CLIP embedding 近重复过滤。
4. 优先使用上游数据的 train split；如果 OVO/StreamingBench 已经使用了某数据源的 val/test，则训练侧要避开相同 split。

### 和两个 benchmark 的对齐方式

对齐 OVO：

- 主线应该从真实上游重叠源开始：`Ego4D / COIN / YouCook2 / HiREST / CrossTask / MovieNet / STAR / Perception-Test / THUMOS`。
- `VStream-QA / CogStream / TimeChat-Online` 虽然不一定是 OVO 直接源文件，但它们复用了 OVO 上游中的多个数据源，可以作为 adapter 成本较低的二级来源。
- 最关键是保留 `query_timestamp / target_timestamp / evidence_timespan / reply_timespan / negative_before`，否则无法变成 StreamWeave online 样本。

对齐 StreamingBench：

- StreamingBench 的视频主要来自人工筛选 YouTube，不是公开 benchmark 合并。
- 因此不追求“同源”，而追求“类别覆盖”：
  - life record：`Ego4D`
  - education / procedure：`COIN / YouCook2 / CrossTask / HiREST`
  - movie / clue reveal：`MovieNet / MMDuetIT`
  - competition / action event：`THUMOS / Perception-Test / STAR`
  - streaming QA / proactive form：`TimeChat-Online / Streamo / VStream-QA / CogStream`
- `StreamingBench` 的 audio/omni-source 暂时不作为第一版训练目标，只对齐视觉和上下文部分。

### 下一步检查清单

对每个候选数据源都要补一张小表：

- 是否已经在本机存在视频和标注。
- license 是否允许训练使用。
- 是否有 train/val/test split。
- 是否有原始 video id / YouTube id，能不能和 OVO/StreamingBench 去重。
- 是否有时间边界：
  - 没有时间边界：只能做普通 QA/SFT，不能进 forward/timing 数据。
  - 有 event/step boundary：可转 `evidence_timespan`。
  - 有 answer/reply window：可直接转 `reply_timespan`。
- adapter 成本：
  - 低：已有 QA + 时间戳。
  - 中：已有 caption/step，需要生成 QA。
  - 高：只有视频或只有离线 QA，需要重新做 temporal localization。

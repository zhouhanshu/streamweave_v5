# 模型复活

本文先只关注 `models/exp1_rl0511_step60` 这一支。目标不是把它直接当最新 base，而是把当时“看起来很有希望”的链路、数据、配置和失败原因钉住，后面可以复现同一条实验，并在同一条线上加 guardrail 继续做。

## 结论

`models/exp1_rl0511_step60` 是 `grpo_rl0511_8gpu_judge` 的 step60 导出模型。按现存脚本和恢复实验记录，它不是从 plain `models/qwen3vl8b_streamweave_sft_answered_full_vllm` 直接训出来的，而是从：

```text
models/qwen3vl8b_streamweave_sft_answered_full_anchor_delta_init_anchor_step200_vllm
```

继续做 GRPO/RL 得到的。

这支值得“复活”的原因是：OVO 1/8 曾到 `64.70`，Realtime 和 Forward 局部有收益；但 full 回评只有 `59.87`，核心问题是 HLD 拒答能力崩了。因此后续复活应复用它的正向部分，而不是原样把 step60 当训练 base。

## 为什么值得复活

这支不是最终好模型，但它暴露了一个有价值的方向：RL 可以把模型从“保守写、保守答”推向“更积极整合 memory 并回答”。相对 GRPO 0509 full，exp1 的局部净增包括：

| Task | RL0511 vs GRPO0509 净增 |
| --- | ---: |
| OCR | +2 |
| ACR | +4 |
| STU | +6 |
| FPD | +5 |
| REC | +10 |

如果只看非 HLD，exp1 和 GRPO0509 大体接近；如果 HLD 能维持 base 的 `65.05`，恢复记录里的反事实 Total 可到 `64.34`，超过 GRPO0509 的 `63.90`。所以复活的核心不是“照抄 step60”，而是保留其 Realtime / Forward 的收益，同时把拒答和边界校准补回来。

## 为什么不能直接当 base

核心失败是 HLD：

```text
HLD 总数: 186
GT: 186/186 都是 Unable to answer
base 对: 121
SFT 对: 111
GRPO0509 对: 112
RL0511 step60 对: 46
```

相对 GRPO0509，HLD 单项净损 `-66`，几乎单独解释 full Total 从可能的高分掉到 `59.87`。

主要根因：

1. `rl_0511.jsonl` 全是 backward，没有 Unable gold，没有 forward/realtime 边界负样本，也没有 counting 专项。
2. `w_success=0.8` 让答具体选项的期望收益高于拒答。
3. 没有 KL，60 步训练后答案分布向“敢答”漂移过头。
4. QA step 同时写 memory 和答题，question 可以倒灌进 delta。
5. 长视频 count 没有复位/校验，出现 REC tail blowup。

Memory 侧也有信号：exp1 full 的 note/frame 是 `14.01%`，高于 GRPO0509 的 `10.36%` 和 SFT 的 `10.32%`。单条 delta 不更长，但 delta 数量更多，说明它更爱写入/保留 memory。

## 复活路线

### 1. 精确复现

目的：确认当前代码和数据恢复后，能不能重现 old exp1 的行为。

做法：

- 复制 `RL/scripts/train_grpo.sh` 为新脚本，例如 `RL/scripts/train_exp1_revive_exact.sh`。
- 新脚本使用新的 `RUN_NAME`，不要写回 `grpo_rl0511_8gpu_judge`。
- `TRAIN_FILE` / `VAL_FILE` 指向恢复数据副本，或先恢复 `dataset2/rl_0511.jsonl`。
- 把 `gen_batch_size` 设回当时 live command 的 `16`。
- 保持 source model 为 `models/qwen3vl8b_streamweave_sft_answered_full_anchor_delta_init_anchor_step200_vllm`。
- checkpoint 先看 step20 / step40 / step60，不要只看最后一步。

验收：

- 能导出 HF/vLLM 模型。
- OVO 1/8 复现到接近 `64.70` 才说明配置基本对齐。
- HLD 仍会大概率崩，这是预期，不要把它误判成环境问题。

### 2. Guardrail 复活

目的：保留 exp1 的正向推力，但补上拒答和边界校准。

建议命名：

```text
exp1_revive_guardrail
```

最小改动：

- source 仍用 `models/qwen3vl8b_streamweave_sft_answered_full_anchor_delta_init_anchor_step200_vllm`。
- 数据从 `rl_0511` 扩成混合数据，至少加入 Unable / false premise / forward boundary / realtime answerability / counting boundary。
- `w_success` 从 `0.8` 降到 `0.5-0.6`。
- success reward 改成 abstention-aware：GT 是 Unable 时，答具体选项应被罚，而不是只给 0。
- 开 `actor.use_kl_loss=true`，`kl_loss_coef=0.001` 起步。
- 先保守训 step20 / step40，不要默认推到 step60。

更结构性的改动：

- 把 memory write 和 answer 拆成两次 forward：先不看 QA 写 delta，再看 frozen memory + QA 答题。
- 不把每步 state 持久写入 memory，只持久保留 anchor/delta，避免 count drift。
- 对 HLD / SSR / CRR / REC 做 task-aware prompt 或 verifier gating。

### 3. 快速门槛

每个 revive checkpoint 在 full OVO 前先过这组 quick check：

| 指标 | 门槛 |
| --- | --- |
| HLD 1/8 | >= 50 |
| HLD Unable rate | >= 0.50 |
| REC bias mean | `[-0.5, +0.3]` |
| REC max over-count | <= 10 |
| non-HLD same-id net vs GRPO0509 | >= -5 |
| note/frame | 8% 到 13% |
| SSR/CRR FP-FN imbalance | < 0.15 |

过不了就不要跑 full，避免浪费 8 卡长评测。

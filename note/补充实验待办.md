## Memory Fidelity Evaluation 草案

目标：设计一个轻量、可自动化的 memory fidelity evaluation，用来证明 StreamWeave 的 Anchor+Delta 图文交错记忆相比纯文本记忆，能够在相同或更少 token 预算下，更忠实地保留原视频前缀的可验证语义信息。

基本设定：

- 输入原视频已观测前缀的 1fps 抽帧。
- 输入不同形式的候选 memory state，将每个 memory state 视为对已观测视频前缀的压缩摘要。
- 使用强 VLM judge，例如 `Qwen3-VL-32B`，基于原始抽帧和候选 memory 做自动评分。
- 所有候选 memory 必须使用同一个视频前缀，禁止使用未来帧信息。
- 对比对象至少包括纯文本 memory baseline 和 StreamWeave Anchor+Delta 图文交错 memory；可选加入同 token 预算纯文本、短预算 Anchor+Delta、oracle caption memory 等参考组。

核心问题：

- memory 是否保留关键对象、动作、状态变化、OCR、颜色、空间关系、数量、身份等重要视觉证据。
- memory 中的描述是否被原视频支持，是否存在编造、细节幻觉或时间顺序错误。
- 在单位 token 预算下，哪种 memory 能保留更多忠实、可验证的视频语义。

Judge 协议：

- Judge 先从原视频 1fps 抽帧中提取 reference evidence，包括关键对象、动作、状态变化、OCR、颜色、空间关系、数量和身份线索。
- Judge 再逐项检查候选 memory 中的显式或隐式 claim 是否被视频证据支持。
- Judge 输出结构化 JSON，包含分数、逐维度简短理由、遗漏证据列表和不被支持的 claim 列表。
- Judge 不奖励冗长描述；同等事实覆盖下，token 更少的 memory 应获得更高信息效率。

评估指标：

- QEVA-style `Coverage`：memory 覆盖原视频关键语义证据的程度，重点看对象、动作、状态变化、OCR、颜色、空间关系、数量和身份。
- QEVA-style `Factuality`：memory 中的描述是否被原视频支持，惩罚无法从帧中验证的细节、错误属性和编造事件。
- QEVA-style `Chronology`：memory 是否保持事件和状态变化的正确时间顺序，惩罚前后颠倒、跨窗口错误合并和错误因果。
- ARGUS-style `Hallucination`：统计 unsupported claims，占比越高越差。
- ARGUS-style `Omission`：统计 missed key evidence，占比越高越差。
- `Token Efficiency`：在 token 预算归一化后计算 fidelity score，例如 `(Coverage + Factuality + Chronology - HallucinationPenalty - OmissionPenalty) / token_count` 或每 1K token 的保真分。

建议产物：

- 输出 JSONL，每行包含 `video_id`、`prefix_end_time`、`memory_type`、`token_count`、`coverage`、`factuality`、`chronology`、`hallucination_rate`、`omission_rate`、`token_efficiency`、`rationale`、`missed_evidence`、`unsupported_claims`。
- 汇总表按 memory type 报告平均分、单位 token 分、不同 token budget 下的曲线和显著性检验。
- 重点抽样展示 OCR、颜色、数量、身份、空间关系和细粒度动作变化 case。

预期结论：

- Anchor+Delta 图文交错记忆不是简单增加记忆容量，而是在相同或更少 token 预算下，以更高的信息效率维持视频语义连续性。
- 相比纯文本压缩，图文交错记忆应降低细粒度视觉信息丢失、时间顺序错误和文本幻觉。

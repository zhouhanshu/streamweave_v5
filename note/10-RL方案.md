## 一句话概括

当前 RL 训练的不是普通离线 QA，而是一个 closed-loop 流式视频 agent：

```text
每个 step 看一段视频帧
  -> 更新 memory
  -> 判断要不要回答
  -> 必要时回答
  -> 环境推进到下一个 step
```

核心是 stepwise GRPPO：每个视频 step 都是训练 row，过程 reward 和答案 reward 分开计算，再在同一视频、同一时间步的多条 rollout 之间做组内比较。

## 训练目标

模型每个 step 的输入：

```text
Memory + QA History + Current frames
```

模型每个 step 的输出：

```xml
<state>...</state>
<answer>...</answer>
<anchor t="..."></anchor>
<delta t="...">...</delta>
```

RL 同时训练四件事：

- memory 是否可靠：`anchor/delta/state` 是否忠实当前帧和历史 memory。
- 回答时机是否正确：该答时答，不该答时 `<answer>` 保持空。
- 答案是否正确：MCQ / text 在标注时间点正确。
- 协议是否稳定：XML tag、时间、parser validity 不能坏。

因此不能只用“最终答案对了”奖励整条轨迹。早期 memory 写错，最后可能仍然猜对；早期 memory 写对，最后因为证据不足乱答也不应该被奖励。

## 数据入口

RL 数据只接受 canonical `query_events` schema。

对应代码：

```text
RL/streamweave_rl/dataset.py
RL/streamweave_rl/env.py
```

每行是一条视频样本：

```json
{
  "dataset": "Streamo-Instruct-465K",
  "video_id": "...",
  "video": "video/...",
  "frames_dir": "video/...",
  "sample_fps": 1.0,
  "frame_count": 81,
  "frame_id_base": 0,
  "query_events": []
}
```

每个 query 必须有：

```text
qid
time
content
answer_type = mcq | text
answer_policy
answer_events
```

MCQ 规则：

```text
query.options 必须存在
answer_events[].gt      选项字母
answer_events[].answer  选项文本
answer_events[].content 推荐写成 "C. blue sponge"
```

Text 规则：

```text
不要写 options
answer_events[].answer 是自然语言标准答案
answer_events[].content 通常等于 answer
```

环境把 `query_events` 转成 timeline：

```text
query          用户问题出现
answer_target 需要评分的回答时刻
```

时间映射：

```text
time 秒
  -> frame_id，按 sample_fps=1.0 和 frame_id_base=0
  -> step，按 frames_per_step=5 分组
```

硬约束：

- 同一 step 最多一个 query。
- 同一 step 最多一个 answer target。
- query 和 answer target 可以在同一 step。
- 帧目录必须存在，RL 不自动抽帧。
- `frame_count` 是有效视频长度，环境按它截断，不直接相信目录里全部图片。
- query 或 answer target 被 `max_steps` / frame truncation 截掉时，该 trajectory 直接 abort。

## Rollout

当前不是拼接全轨迹训练，而是 no-concat stepwise rollout。

主链路：

```text
StreamWeaveAgentDataset
  -> StreamWeaveAgentLoop.run
  -> StreamWeaveRLEnv.reset
  -> 按 frames_per_step 切视频
  -> 每个 step 渲染 fresh prompt
  -> vLLM 生成当前 XML
  -> env.evaluate_attempt(raw, repair=True)
  -> raw XML 算 reward
  -> repaired action 推进 memory
  -> 返回 AgentLoopOutput
```

为什么必须 fresh prompt：

- 当前 step 的 prompt 依赖上一步生成的 memory。
- QA History 会随着 query 出现而变化。
- 图片窗口会随视频推进变化。
- 所以不能预先把整条轨迹拼成一个普通 chat transcript。

每个 step row 保留：

```text
group_idx  同一个原始样本 / 同一个视频问题组
traj_idx   同一 group 下第几条 rollout
turn_idx   当前视频 step
last_turn  是否最后一步
```

GRPPO 比较口径：

```text
同一个 group_idx
同一个 turn_idx
不同 traj_idx
```

也就是同一个视频在同一个时间步的多个采样之间比较，而不是把不同视频或不同时间步混在一起。

## Answer Supervision

当前使用：

```text
grppo_answer_event_mode=timeline
```

代码：

```text
RL/streamweave_rl/env.py::_timeline_grppo_answer_supervision
```

每个 step 的 answer supervision 有三类：

| kind | 含义 | 训练要求 |
|---|---|---|
| `none` | 当前没有可评分回答事件 | 不给 answer reward |
| `silence` | 已有 query，但现在不该回答 | `<answer>` 应为空 |
| `answer` | 当前 step 是 answer target | `<answer>` 必须回答 |

触发规则：

- 当前 step 有 answer target 且 label 要求回答：`kind=answer`。
- 当前 step 有 query 但不是立即可答：`kind=silence`。
- 当前没有 query/target，但模型无故输出非空 `<answer>`：构造 `unprompted_answer` 的 silence label。
- parser invalid 时 answer reward 直接无效或为 0。

`silence` 的意义是：问题已经进入 QA History，但视频证据还不充分，模型应该继续看视频，而不是提前猜。

## Judge

当前 judge prompt：

```text
streamweave_grppo_judge_v1
```

代码：

```text
RL/streamweave_rl/judge.py
RL/streamweave_rl/judge_prompts.py
```

和旧 proposal 的差异：

- 旧 proposal 设想一个 answer-aware prompt 同时返回过程分和答案分。
- 当前实现拆成两段：
  - process judge：每个 step 都跑，评估 memory / state / anchor / delta。
  - answer judge：只在 `silence` 或 `answer` supervision step 跑，评估 `<answer>` 是否满足当前要求。

### Process Judge

process judge 返回四个维度：

```text
delta_groundedness
anchor_keyframe
semantic_alignment
state_groundedness
```

每个维度由 checklist 打分，单项是 `1 / 0.5 / 0`。最终过程分：

```text
grppo_judge_step_reward = 2.0 * mean(process dimensions)
```

四个维度含义：

| 维度 | 看什么 |
|---|---|
| `delta_groundedness` | delta 是否描述真实可见变化，是否保留 query 相关细节，是否避免把问题/选项当作视觉事实 |
| `anchor_keyframe` | 每步最多一个 anchor；时间合法；body 为空；首窗口有 `0.0-1.0` anchor |
| `semantic_alignment` | state、answer、anchor、delta 是否互相一致，是否符合当前帧顺序和历史 memory |
| `state_groundedness` | state 是否基于可见证据，是否正确判断问题范围和回答/沉默决策 |

### Answer Judge

answer judge 只返回：

```text
answer_reward = 0.0 | 1.0
```

它看到的是环境渲染后的 requirement，而不是原始标注 JSON。

silence 示例：

```text
Requirement: the model should not answer now; <answer> should be empty.
```

answer 示例：

```text
Requirement: the model must answer now. Reference answer: C. red
```

注意：`grppo_answer_correctness` 主要是 rule scorer 指标和 trajectory target 统计，不直接覆盖 LLM judge 的 answer reward。

## Reward

### Step Process Reward

当前 step reward：

```text
grppo_step_reward =
  grppo_process_weight * grppo_judge_step_reward
  + grppo_format_weight * grppo_format_score
  + grppo_note_frequency_weight * grppo_note_frequency_score
```

exp9_24 默认：

```text
grppo_process_weight = 0.7
grppo_format_weight = 0.15
grppo_note_frequency_weight = 0.15
```

字段说明：

- `grppo_judge_step_reward`：LLM judge 给出的过程分。
- `grppo_format_score`：raw XML 质量。
- `grppo_note_frequency_score`：约束 anchor 频率，避免长时间无 anchor 或一轮多个 anchor。

代码里仍有历史名 `note/bridge`，当前协议层面对应的是 `anchor/delta`。

### Answer Reward

answer step：

```text
有非空 answer 且 judge 判对 -> 1
否则 -> 0
```

silence step：

```text
模型输出非空 answer -> 0
模型保持沉默且 judge 判满足 requirement -> grppo_silence_reward_value
```

exp9_24 默认：

```text
grppo_silence_reward = true
grppo_silence_reward_value = 0.25
```

### Answer Credit

答案 reward 不只影响 answer target 当前 step，还会向前传播：

```text
answer_credit_t = answer_reward_t + beta * answer_credit_{t+1}
```

exp9_24 默认：

```text
beta = grppo_answer_decay = 0.5
```

这样早期 memory、state、沉默决策可以因为未来答对获得 credit；未来 judge 出错时，对应 credit 段会被标成 invalid，避免错误梯度。

## Advantage

当前 estimator：

```text
streamweave_stepwise_grppo
```

代码：

```text
RL/streamweave_rl/advantage.py
```

计算流程：

```text
step_advantage_t =
  center(grppo_step_reward_t within (group_idx, turn_idx))

answer_advantage_t =
  center(answer_credit_t within (group_idx, turn_idx))

grppo_advantage_t =
  grppo_step_weight * step_advantage_t
  + grppo_answer_weight * answer_advantage_t
```

exp9_24 默认：

```text
grppo_step_weight = 1.0
grppo_answer_weight = 0.5
grppo_norm_by_std = false
grppo_min_std = 0.05
```

含义：

- 默认只做 mean-center，不除以 std。
- step reward 和 answer credit 分开归一化。
- 某个 component 方差太小时，只关掉这个 component。
- 如果另一个 component 有足够方差，这个 row 仍可训练。

最后写入 token：

```text
advantages[row_tokens] = grppo_advantage_t * response_mask
returns[row_tokens] = grppo_reward_t * response_mask
```

## GRPPO Filter

代码：

```text
RL/verl/verl/trainer/ppo/ray_trainer.py::_maybe_apply_streamweave_grppo_filter
```

filter 发生在：

```text
rollout 完成
  -> 计算 GRPPO component / valid flags
  -> filter batch
  -> old_log_prob
  -> actor update
```

流程：

1. 可选 forced-answer postprocess。
2. 对完整 batch 计算 `grppo_answer_credit`、component advantages 和 valid flags。
3. 按 `(group_idx, turn_idx)` 统计 step reward / answer credit 的 std。
4. step component 或 answer component 至少一个有足够方差，则该 cohort 可训练。
5. 如果全 invalid，保留原 batch，但 advantage 会被置零，避免空 batch 破坏训练。

exp9_24 默认：

```text
grppo_filter_groups.enable = true
grppo_filter_groups.min_std = 0.05
grppo_forced_answer_postprocess_enable = false
```

注意：

```text
algorithm.filter_groups.enable=false
```

这是关闭旧 trajectory-level DAPO filter；当前 step-level filter 走：

```text
algorithm.grppo_filter_groups
```

## Judge Error

judge error 不能产生错误梯度。

当前处理：

- `judge_status in {"error", "aborted"}` 的 row，process component 无效。
- answer credit 会沿同一 trajectory 向前传播 invalid。
- 如果未来某 step judge 出错，之前依赖未来 answer credit 的 row 也不使用这段 answer component。
- debug dump 会记录 `judge_error` 和 `judge_raw_response_preview`。

关键字段：

```text
judge_status
judge_error
judge_raw_response_preview
grppo_step_signal_valid
grppo_answer_signal_valid
grppo_update_signal_valid
```

## Validation

训练主 advantage 不直接使用 trajectory reward，但 validation 仍需要 trajectory 级观测指标。

当前 exp9_24：

```text
stepwise_validation_score_key = grppo_target_trajectory_score
```

计算：

```text
grppo_target_trajectory_score =
  grppo_target_answer_weight * target_answer_reward
  + grppo_target_format_weight * format_mean
```

exp9_24 默认：

```text
grppo_target_answer_weight = 1.0
grppo_target_format_weight = 0.0
```

因此 validation 主分主要看标注 answer target 的答对率，不把 process judge 混进 validation 主分。

## 和旧方案的主要差异

1. 当前协议是 `<state>/<answer>/<anchor>/<delta>`，不是旧 V3 的 `<eta>`。
2. 当前训练是 no-concat stepwise RL，不是拼接全轨迹的普通 SFT/RL。
3. 当前 answer judge 是独立 answer-only prompt，不再让一个 answer-aware prompt 同时返回五个分数。
4. 当前 `grppo_answer_correctness` 是指标，不直接覆盖 LLM judge 的 answer reward。
5. exp9_24 默认 `grppo_answer_decay=0.5`，旧 proposal 初始建议过 `0.7`。
6. exp9_24 默认 `grppo_silence_reward_value=0.25`。
7. 当前 validation 主分用 `grppo_target_trajectory_score`，不是旧 `trajectory_score`。
8. 当前代码有 judge error valid mask 和 prefilter debug dump，旧 proposal 没展开工程细节。

## 当前边界

- 数据必须先保证 step 内 query/answer target 不冲突。
- text answer 可以用 judge 评分，但质量依赖 Gemini judge 稳定性。
- Gemini judge JSON parse 失败会让相关 row 无效，不能被当作模型负样本。
- 当前没有独立 process reward model，process score 依赖 LLM judge。
- 当前训练重点是流式 answerability、semantic drift、多次更新和 memory quality。
- 后续新增数据应优先补 evidence gate、unable、语义漂移、多 query 同视频，而不是普通离线 QA。

## 常见误解

| 误解 | 正确口径 |
|---|---|
| RL 是把整条视频拼成一个长 prompt 训练 | 不是。每个 step 都重新渲染 prompt，按环境状态推进 |
| 只要最后答对就奖励全轨迹 | 不是。step process reward 和 answer credit 分开 |
| `frames_per_step=1` 更细 | 当前主线不是这个口径，训练和清洗按 `frames_per_step=5` |
| 多 query 同视频都有逻辑依赖 | 不一定。CogStream 多 query 主要是共享视频，降低训练开销 |
| silence 是负样本 | 不是。silence 是证据不足时的正确行为，可以得到正 reward |
| yaml 里的默认就是当前实验 | 不一定。当前 exp9_24 主要靠 `train_exp9_24.sh` 覆盖 |

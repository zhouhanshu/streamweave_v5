## 当前判断

后续主要训练数据将以 `ActivityNet_Captions` 为视频源，以 `VideoXum` 的摘要和关键帧标注作为重要监督信息。

`VideoXum` 本身不提供原始视频，只提供：

- 视频级摘要标注
- 视觉摘要标签
- BLIP / CLIP 预提取特征

因此现在已经具备让 VLM 直接读取 ActivityNet 视频、并用 VideoXum 标注对齐监督的基础数据。

但需要注意：`VideoXum` 的 `vsum_onehot` 不是 QA evidence frame，而是视频级、query-free 的视觉摘要帧。它只能提供关键帧先验，不能直接定义问题、答案和最早可回答时间。

构造规则：

- 平均后的逐帧分数写入 `key_frame_scores`。

## VideoXum-StreamQA 数据合成方案

目标输出：

```text
exp2/data/videoxum_streamqa/
```

建议文件：

```text
raw_videoxum_index.jsonl
keyframe_segments.jsonl
atomic_facts.jsonl
qa_candidates.jsonl
qa_verified.jsonl
teacher_traces.jsonl
sft_steps.jsonl
reports/verification_stats.json
reports/human_audit_sample.jsonl
```

**输入**
- 视频（1fps 抽帧，帧 id = 秒数）
- 关键帧列表，如 `[12, 27, 45, 63, 88]`（由外部检测器提供，标注语义变化点）

**输出**
- 每个视频先生成 6 条候选流式 QA，经过过滤后最终保留 2 条高质量样本

### Step 1：构造时间窗口

#### 关键帧窗口

对每个关键帧 k，构造局部窗口 `[k-2, k+2]`（前后各 2 秒）：

```json
{
  "window_id": "kw_003",
  "source_keyframe_id": 45,
  "time": [43, 47],
  "frames": ["000043.jpg", "...", "000047.jpg"]
}
```

#### 普通窗口

剩余区间（不属于任何关键帧窗口）均匀切分，每段严格为 5 秒；尾段不足 5 秒时保留为一个短普通窗口。

**关键帧窗口合并规则**：若两个相邻关键帧 k1、k2 之间的间距大于 0 且小于 8，则将两个关键帧窗口之间的区间合并为一个窗口；若两帧相连（间距 ≤ 4，即窗口直接重叠），则直接合并为一个关键帧窗口，不产生普通窗口。

### Step 2：生成描述

#### 普通窗口

正常生成 caption，不刻意丢失细节。

#### 关键帧窗口

生成三类内容：

- **前后 2 秒 caption**：正常描述 `[k-2, k)` 和 `(k, k+2]`。
- **关键帧窗口整体 caption**：对 `[k-2, k+2]` 整段的正常描述，用于文字验证时的输入。
- **关键帧细粒度事实**（`fine_visual_facts`）：必须来自图片，不能来自文字推断。

> **关键约束**：`fine_visual_facts` 和 `window_caption` 严格分开。过滤验证时，文字验证只用 `window_caption`，不能将 `fine_visual_facts` 喂给文字验证模型。

### Step 3：选取关键帧并生成 QA

每个视频总共生成 6 题，不是每个关键帧各生成一组题；后续过滤保留 2 题。

#### 三类问题的时间关系

| 类型 | query_time | answer_time | 说明 |
|------|-----------|-------------|------|
| realtime | 随机取 k 附近（evidence_interval 内） | k | 证据即时可见 |
| forward | 随机取 [k_prev, k) 区间内某秒 | k | 提问时证据未出现，等到 k 才能答 |
| backward | 随机取 (k, k_next] 区间内某秒 | k | 证据在 k，提问在 k 之后，基于历史回答 |

`k_prev` / `k_next` 指被选中用于生成 QA 的关键帧集合中当前目标关键帧的前/后一个，不是原始关键帧列表中的所有关键帧。

对于 `forward` 和 `backward`，目标关键帧与对应前后关键帧的时间差必须满足：

```text
10s < abs(k - k_prev_or_next) < 60s
```

**query_time 随机规则**：
- realtime：在 `[k-1, k+1]` 内随机取整秒
- forward：在 `[k_prev+1, k-1]` 内随机取整秒；仅当 `10s < k - k_prev < 60s` 时生成
- backward：在 `[k+1, k_next-1]` 内随机取整秒；仅当 `10s < k_next - k < 60s` 时生成

#### 问题子类型

- **realtime**：`realtime_object` / `realtime_attribute` / `realtime_ocr` / `realtime_spatial`
- **forward**：`forward_event` / `forward_state_change`
- **backward**：`backward_object` / `backward_action` / `backward_state`

#### 答案格式策略

- 每个视频总共生成 6 条 QA，优先覆盖不同关键帧和不同问题类型。
- 尽量包含 short-answer 和 multiple-choice。
- yes/no 最多 1 条，除非其它题型无法构造。
- multiple-choice 为 4 选 1，干扰项优先从**其他关键帧**的 `fine_visual_facts` 中采样；若不足，再由模型生成合理但不正确的干扰项。
- yes/no 的 no 问题中，错误对象优先来自视频其他关键帧的真实事实；若没有可用事实，再由模型生成合理但不正确的错误对象。

### Step 4：生成 Prompt

每次围绕一条视频生成 QA，给模型的输入包含全视频 timeline caption、候选目标关键帧、每个目标关键帧的 `fine_visual_facts`，以及其他关键帧 facts。模型需要在整条视频内总共生成 6 条 QA。（详见 `docs/数据合成0427.md` 中的完整 prompt 模板。）

### Step 5：三类过滤规则

过滤目标：保留"纯文字答不出、看图才能答"且时间逻辑正确的样本。最终每个视频保留 2 题。

#### 答对/答错判断方式

| 格式 | 判断方式 |
|------|---------|
| multiple-choice | 直接字符串匹配选项字母 |
| yes/no | 直接字符串匹配 |
| short-answer | 大模型打分（0/1），prompt 要求判断语义是否等价 |

#### realtime 过滤

保留条件：`text_only_correct = false` AND `image_text_correct = true`

#### forward 过滤

保留条件：`text_only_correct = false` AND `prefix_before_query_answerable = false` AND `image_text_correct_at_answer = true` AND `query_time < answer_time`

#### backward 过滤

保留条件：`text_only_correct = false` AND `history_to_query_answerable = true` AND `answer_time < query_time`

历史图像优先使用已选关键帧，默认最多输入 8 张历史关键帧图，必要时把该 QA 的历史证据帧也加入。

## SFT 数据合成格式

SFT 数据采用 **step-level 中间格式**，不直接把数据唯一保存成某个 trainer 的 `messages` 格式。每条样本对应一次 V3 rollout 的单步决策：

```text
input  = memory_before + qa_history + current_frames
target = 当前步教师模型输出的 XML
```

现有 `streamweave_v3` 推理框架每一步实际需要的也是这几类信息：`rollout.py` 会把 `memory / qa_history / current frames` 渲染成 prompt，模型输出 XML 后再解析并更新 memory。因此 SFT step 样本只需要完整保存这些运行时状态，以及教师输出。

### 核心字段

```json
{
  "sample_id": "v_xxx_qa_000_step_003",
  "video_id": "v_xxx",
  "qa_id": "v_xxx_k45_forward_001",
  "question_type": "forward",
  "step_index": 3,
  "step_start": 20.0,
  "step_end": 25.0,
  "memory_before": [],
  "qa_history": [],
  "current_frames": [],
  "raw_teacher_xml": "",
  "target_xml": "",
  "quality": {}
}
```

字段说明：

- `memory_before`：当前 step 调模型之前的 memory，不能保存当前 step 输出之后的 memory，避免信息泄漏。
- `qa_history`：当前 step 调模型之前已经进入系统的问答日志，按时间顺序保存。
- `current_frames`：当前 step 输入给模型的图片帧，必须保存图片路径。
- `raw_teacher_xml`：教师模型原始输出，用于审计和排查。
- `target_xml`：最终用于 SFT 训练的 assistant target。若 postprocess 只修复了小的时间戳或 frame id 问题，可以把修正后的干净 XML 放这里；如果修复过多或质量不合格，样本应直接丢弃。
- `quality`：保存 parser、时间合法性、answer 时机、note/frame 对齐、teacher 过滤分数等质量信息，供后续筛选和统计使用。

### 图片路径字段

训练 dataloader 必须能从 JSONL 重新加载图片，因此不能只保存 `<image/>` 占位符。至少保存两类图片路径：

```text
current_frames[].image_path
memory_before[].image_path   # 仅未过期 note 需要
```

建议结构：

```json
{
  "memory_before": [
    {
      "type": "bridge",
      "t": [0.0, 16.0],
      "text": "The person prepares ingredients at the counter."
    },
    {
      "type": "note",
      "t": [16.0, 17.0],
      "image_path": "video_frame/v_xxx/000016.jpg",
      "image_available": true
    },
    {
      "type": "note",
      "t": [4.0, 5.0],
      "image_path": "video_frame/v_xxx/000004.jpg",
      "image_available": false
    }
  ],
  "current_frames": [
    {
      "frame_id": 1,
      "global_frame_id": 20,
      "t": [20.0, 21.0],
      "image_path": "video_frame/v_xxx/000020.jpg"
    },
    {
      "frame_id": 2,
      "global_frame_id": 21,
      "t": [21.0, 22.0],
      "image_path": "video_frame/v_xxx/000021.jpg"
    }
  ]
}
```

约定：

- `current_frames.frame_id` 是当前 step 内局部编号，从 1 开始，必须对应 `target_xml` 里的 `<note frame="N"></note>`。
- `global_frame_id` 是视频全局 1fps 帧号，对应真实帧文件。
- `image_path` 优先使用相对路径；数据集配置中单独记录 `image_root`。
- 过期 memory note 只在 prompt 中渲染为 `<note t="..."></note>`，不加载图片，`image_available=false`。

### target_xml 定义

`target_xml` 就是 SFT 的监督目标，也就是 assistant 应该输出的 XML。它和 `raw_teacher_xml` 的区别是：

```text
raw_teacher_xml = 教师模型原始输出
target_xml      = 通过程序性检测后被接受的教师原始 XML
```

示例：

```xml
<eta>45.0</eta>
<answer></answer>
<bridge t="20.0-25.0">The person continues preparing items on the counter.</bridge>
<note t="24.0-25.0" frame="5"></note>
```

训练时渲染为：

```text
user:      V3 prompt(memory_before + qa_history + current_frames)
assistant: target_xml
```

`target_xml` 只允许包含协议 XML，不写解释性文字。`eta` 使用视频内绝对秒数；`answer` 在证据不足时必须为空；`note frame` 只能引用当前 step 的局部 `frame_id`，不能引用全局帧号。`note` 必须使用成对标签 `<note ...></note>`，自闭合 `<note .../>` 视为非法。SFT 合成阶段不能对教师输出做程序性改写；不合法就 retry 或丢弃。

### SFT teacher 生成策略修正

SFT 数据合成不能直接复用 inference 阶段的“强修复后处理”作为最终监督目标。推理时为了让 rollout 不崩，可以修复 note 时间戳、clamp bridge 区间，或者在 open-tail 场景下把第一条 bridge 的起点改成上一条 bridge 的起点并覆盖旧 bridge。但训练数据的 `target_xml` 如果来自这种自动改写，会隐藏教师模型的协议错误，并可能造成语义不连贯。

更合理的 SFT teacher 策略：

- `raw_teacher_xml` 始终保存教师模型原始输出。
- `target_xml` 只接受教师模型自己生成且通过校验的输出。
- 不允许做任何程序性改写，包括去掉 `<think>`、补标签、改 note frame、修 note 时间、clamp bridge、改 open-tail bridge 起点、重排事件顺序、统一排版等。教师输出不符合协议时只能 retry；多次失败则丢弃该 step。
- 一旦检测到 `open_tail_start_mismatch`、bridge 越界、note frame/time 不匹配、缺失 `<eta>/<answer>`、输出 note 未命中标注帧等问题，应构造 retry feedback，让教师模型重新输出。

open-tail 场景尤其需要 retry，而不是自动修正。若 Memory 最后一条是：

```xml
<bridge t="A-B">...</bridge>
```

且当前 step 需要继续写 bridge，则教师输出必须自己写成：

```xml
<bridge t="A-C">...</bridge>
```

其中 `C` 是当前 step 的结束时间或当前覆盖区间结束时间。只有教师自己把旧 bridge 的语义自然延展到 `A-C`，这条样本才适合进入 SFT。否则即便程序把 `B-C` 改成 `A-C`，文本仍可能只描述 `B-C`，训练会学到不一致目标。

当前数据合成还有关键帧标注，因此 note 选择不应完全交给教师自由决定。SFT 合成时应把当前 step 内可用的标注关键帧作为约束传给 teacher，并且只用于 teacher 生成与 validator 检测，不能暴露给最终训练 prompt。要求：

- 如果当前 step 覆盖了 GT keyframe / evidence frame，`<note>` 必须优先选择对应局部 `frame_id`。
- 如果没有 GT keyframe，才允许 teacher 根据显著状态变化自行选择或不选择 note。
- teacher 仍负责写 `<bridge>` 文本和 `<answer>/<eta>`，但 `<note frame="N"></note>` 应由标注约束主导。
- 如果 teacher 输出的 note 不在允许 frame set 内，应 retry；多次 retry 失败则丢弃该 step。
- `anchor_constraints` 可以保存在中间表示中用于审计，但 LlamaFactory 导出时不得进入 `messages[0].content`。

训练 prompt 也应和 teacher prompt 解耦。teacher prompt 可以很详细，用于稳定合成高质量 XML；最终 SFT 使用的 user prompt 应是更短的 deployment prompt，只保留模型推理时真正会看到的协议、memory、QA history 和 current frames。中间表示需要同时保存：

```text
teacher_prompt_text    # 用于审计教师生成
train_prompt_text      # 用于导出 LlamaFactory
target_xml             # 通过校验的教师 XML
```

这样可以避免模型在 SFT 时学习过长的 teacher-only 指令，同时保留完整审计链路。

最终导出原则：

```text
teacher_prompt = 详细合成规则 + GT keyframe 约束 + retry feedback
train_prompt   = 短部署提示词 + memory_before + qa_history + current_frames
assistant      = 未被程序改写、通过严格检测的 target_xml
```

主要模块职责：

- `io_utils.py`：统一处理 JSON/JSONL、帧路径、帧存在性检查。现在帧缺失不会静默跳过，会报出具体缺失帧。

- `gen_qa.py`：基于 timeline captions、关键帧 facts 和证据图生成 8 条多选 QA 候选；保存 raw QA response 和 retry attempt。
- `filter_qa.py`：过滤 QA；当前不再做“题干+选项本身能否猜答案”的 leakage auditor。
- `export_annotations.py`：把过滤结果导出成最终 flat annotation。
- `run_pipeline_parallel.py`：按视频粒度并行跑完整 pipeline，完成一条视频就立即落盘 checkpoint。

### Prompt 与过滤策略变更

已按当前需求改成纯多选题生成。QA 生成要求：

- 每题 4 个选项 A/B/C/D。
- 正确答案唯一。
- 不生成 open-ended / yes-no。
- 不允许 `unknown` / `cannot be determined` / `none of the above` 等逃逸选项。
- 题目优先基于细粒度视觉事实：物体、属性、文字、颜色、空间关系、动作、姿态、计数、局部变化。

过滤阶段当前保留：

- MCQ 基本质量检查。
- realtime 的 text timeline 泄漏检查。
- realtime / forward 的证据图可答检查。
- forward 的 ask_time 之前历史不可答检查。
- backward 的历史可答检查。
- 时间关系检查。

过滤阶段已经移除：

- “只看 question + options 是否能猜中答案”的题干/选项泄漏 auditor。

异常处理：

- 单条 QA 在 filter verifier 中出错时，直接丢掉该 QA。
- 丢掉的 QA 写入 `dropped_qa` / `dropped_count`。
- 这类 QA 级错误不再算作视频级 `errors`，也不会进入 `verified_qa` 或 `accepted_qa`。

### QA retry 与 raw response 记录

`gen_qa.py` 已增加：

- `--qa-max-attempts`，默认 3。
- `--qa-raw-response-chars`，默认保存 12000 字符，`-1` 可保存完整 raw response。
- 每次 QA 生成 attempt 会写入 `qa_generation_attempts`，记录 raw response、解析候选数、合法候选数、被拒原因。

这解决了之前 Gemini 正式 QA 调用返回空或 schema 不规范时无法定位的问题。

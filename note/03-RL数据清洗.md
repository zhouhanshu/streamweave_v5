# RL 数据清洗

本文记录当前 RL 数据从 `dataset2` 层原始转换标注，到统一 `query_events` schema，再到 Gemini 难度清洗和最终训练文件的完整链路。视频下载、抽帧和 `dataset2/<dataset>/video/` 的生成见 `01-数据准备.md`；训练命令见 `05-RL训练.md`。

## 一句话链路

```text
dataset2 层原始转换标注
  CogStream/annotations.jsonl
  Streamo-Instruct-465K/annotations.jsonl
  NeXTVideo / LLaVA-Video / PerceptionTest 的旧 rl_tmp / rl_0512
  -> query_events 统一 schema
  -> 0514 核心池：pre / CogStream / Streamo unable / Streamo one / Streamo multi
  -> 0515 yes/no evidence-sufficient 派生
  -> Gemini 4 次难度清洗，写回 eval_pass_rate
  -> 按 answer_event 难度抽样 + 500 条 pre
  -> dataset2/rl_0516_filter.jsonl
```

这里的“原始标注”指 `dataset2` 里保留下来的统一转换标注，不是官方数据包里的原始 JSON。比如 `dataset2/CogStream/annotations.jsonl` 已经绑定了本地 `video_id`、`frames_dir`、`frame_count`、`sample_fps=1.0`、`question/answer/gt`、时间字段和 `source_*` 追溯字段。

## Canonical Schema

每行 JSONL 是一条视频样本，顶层只保留视频信息和 `query_events`：

```json
{
  "dataset": "Streamo-Instruct-465K",
  "video_id": "...",
  "video": "video/...",
  "frames_dir": "video/...",
  "sample_fps": 1.0,
  "frame_count": 81,
  "frame_id_base": 0,
  "query_events": []
}
```

每个 query 是唯一 QA 标注入口：

```json
{
  "qid": "q0",
  "time": 31.0,
  "content": "Question...\nOptions:\nA. ...\nB. ...",
  "answer_type": "mcq",
  "answer_policy": "answer_when_asked",
  "options": ["...", "..."],
  "answer_events": []
}
```

`answer_type` 只允许：

```text
mcq
text
```

MCQ answer event：

```json
{
  "time": 81.0,
  "gt": "C",
  "answer": "blue sponge",
  "content": "C. blue sponge"
}
```

Text answer event：

```json
{
  "time": 18.0,
  "answer": "The person is cutting vegetables.",
  "content": "The person is cutting vegetables."
}
```

硬约束：

- `time` 必须是数字秒。
- `sample_fps=1.0`，`frame_id_base=0`。
- `frame_count` 是有效视频长度，不自动取真实帧目录长度。
- MCQ 必须有 `options`，answer event 必须有 `gt`。
- text 不能写 `options`，answer event 不写 `gt`。
- 同一 step 内最多一个 query、最多一个 answer target。
- 当前训练按 `frames_per_step=5` 分 step，所以数据阶段要避免 5 秒窗口内多个 query 或多个 answer target。
- 帧目录必须能在 `dataset2/{dataset}/{frames_dir}` 找到。

### Streamo query_events

`build_query_event_annotations.py` 做的事情：

- 从 Streamo `annotations.jsonl` 的 `qa_list` 里读 MCQ。
- 按 `source_file / source_dataset / source_video_path / ask_time / question / options` 聚合。
- 一个 query 可以有多个 `answer_events`，表示同一问题在视频推进中答案发生变化。
- MCQ `content` 会把 options 写进问题文本。
- 多答题会加 `Update your answer when the video evidence changes.`。

### 0514 基础过滤

`filter_query_events_0514.py` 的规则：

```text
min_video_seconds = 20
max_video_seconds = 110
min_answer_gap_seconds = 7
max_frame_tail_seconds = 3
```

过滤/修正：

- 所有 query / answer time 四舍五入到 1fps 秒。
- 只保留 20s 到 110s 的有效视频。
- 超过 110s 但 query 和第一个 answer 都在 110s 前，可以截断到 110s。
- 截断时保留 110s 前最后一个 answer。
- answer 间隔小于 7s 的 QA 过滤。
- `frame_count` 必须不小于最后一个 answer time，允许大 2-3s。
- options 重复或近重复直接丢弃。

```text

truncated:    331

```

drop reason：

### 分成 unable / one / multi

`split_streamo_rl_0514.py` 把 3544 条 filtered Streamo 分成：

```text
unable: 925
one:    2318
multi:  301
```

三类含义：

- `multi`：一个 query 有多个 answer_events，保留为 semantic drift / update_when_changed。
- `one`：普通单问单答 MCQ。
- `unable`：从符合条件的单问单答样本派生，构造成“当前视频证据不足，应该选 Unable”。

unable 派生规则：

- 原样本必须只有一个 answer。
- 原 query time 至少 20s。
- 原 answer time 和 query time 至少间隔 20s。
- 视频 `frame_count` 截断到 query time。
- 将一个非 GT 选项替换成 `Unable to answer from the video so far`。
- GT 改成这个 unable 选项。
- 问题只加 `Based on the video so far, ...`，不额外提示“如果证据不足就选 unable”。

### 标准化

`normalize_rl_schema_0514.py` 做的事情：

- 只保留 canonical top-level / query / answer 字段。
- 所有时间统一四舍五入成整数秒。
- MCQ / text 字段严格分离。
- 对 `update_when_changed` 的问题前加：

```text
Please answer the following question based on the video content. You may update your answer multiple times.
```

## 来源 4：yes/no evidence sufficient

文件：

```text

```

脚本：

```text
dataset2/build_rl0515_yesno.py
```

来源：

```text
Streamo-Instruct-465K/rl_0514_one_normalized.jsonl
```

构造目标是 OVO-CRR-style 证据充分性判断：同一个原问题拆成两个独立样本，一个在证据不足时回答 `No`，一个在证据出现后回答 `Yes`。

筛选条件：

- 来源是 Streamo one 中未进入 `rl_0515_train/val` 的样本。
- `answer_policy=update_when_changed`。
- 单 query、单 answer、MCQ。
- query time 和 answer time 间隔 15s 到 30s。

构造方式：

```text
No 样本：截断到原 query time，问证据是否足够，答案 No
Yes 样本：截断到原 answer time，问同一个问题证据是否足够，答案 Yes
```

prompt 模板：

```text
You're responsible for answering questions based on the video content.
The following question is relevant to the latest frames, i.e. the end of the video.
{question}
Decide whether the existing visual content, especially the latest frames near the end of the video, provides enough information for answering the question.
Answer only with "Yes" or "No".
Do not include any additional text or explanation in your response.
```

当前规模：

```text
available_candidates:  216
pairs:                 200
rows:                  400
answer_counts: Yes 200, No 200
```


## 校验口径

当前最终文件已按这些规则检查：

- 每行都有非空 `query_events`。
- 除 pre 外，选中的 answer_events 都有 `eval_runs/eval_correct/eval_pass_rate`。
- `frames_per_step=5` 下没有同 step 多 query / 多 answer 冲突。
- `query_time <= answer_time`。
- `answer_time <= frame_count`。
- MCQ 都有 `options` 和 `gt`。
- text 不写 `options` / `gt`。
- 帧目录存在。

## 关键结论

- 当前 RL 训练主线不是简单合并数据，而是保留中等难度 answer_event，少量混入全错和全对，再额外补 500 条 pre MCQ。
- CogStream 负责提供同一视频多 query 的 realtime text QA，可以降低训练推理开销。
- Streamo 负责提供 MCQ、semantic drift、unable 和 evidence-sufficient yes/no。
- `rl_0514_pre` 负责保留旧 NeXTVideo / YouTube / PerceptionTest 的末尾回答 MCQ，但没有参与 Gemini 难度清洗。
- 当前最重要的可追溯文件是 `dataset2/0516data/rl/*` 和 `dataset2/rl_0516_filter.summary.json`。

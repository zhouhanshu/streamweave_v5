## Stepwise 链路

核心代码：

```text
RL/streamweave_rl/dataset.py
RL/streamweave_rl/env.py
RL/streamweave_rl/agent_loop_stepwise.py
RL/streamweave_rl/rewards.py
RL/streamweave_rl/judge.py
RL/streamweave_rl/advantage.py
RL/verl/verl/trainer/ppo/ray_trainer.py
```

流程：

```text
StreamWeaveAgentDataset
  -> StreamWeaveAgentLoop
  -> StreamWeaveRLEnv.reset()
  -> 按 frames_per_step=5 切视频
  -> 每个 step 渲染 Memory + QA History + Current frames
  -> vLLM 生成 XML
  -> raw XML 计 reward
  -> repaired action 推进 memory
  -> stepwise rows 回传 verl
```

每条 step row 保留：

```text
group_idx
traj_idx
turn_idx
last_turn
```

当前原则：

- 不做 trajectory concat。
- 一条视频一个 trajectory，但会展开成多条 step-level rows。
- 同一原始样本同一 step 的多 rollout 用 `(group_idx, turn_idx)` 做组内比较。
- reward 看 raw XML；memory 推进可以用 repaired action。
- query / answer target 来自 `query_events` timeline。
- 训练数据已按 `frames_per_step=5` 检查，同 step 不应有多个 query 或多个 answer target。

## Reward 口径

当前主 estimator：

```text
algorithm.adv_estimator=streamweave_stepwise_grppo
critic.enable=false
algorithm.use_kl_in_reward=false
algorithm.filter_groups.enable=false
```

`algorithm.filter_groups.enable=false` 关闭的是旧 trajectory-level filter；当前 step-level filter 走：

```text
algorithm.grppo_filter_groups.enable=true
algorithm.grppo_filter_groups.min_std=0.05
```

当前 exp9_24 GRPPO 参数：

```text
grppo_answer_event_mode=timeline
grppo_answer_decay=0.5
grppo_step_weight=1.0
grppo_answer_weight=0.5
grppo_norm_by_std=false
grppo_min_std=0.05
grppo_forced_answer_postprocess_enable=false
```

process reward：

```text
grppo_step_reward =
  0.7  * grppo_judge_step_reward
  +0.15 * grppo_format_score
  +0.15 * grppo_note_frequency_score
```

process judge 维度：

```text
delta_groundedness
anchor_keyframe
semantic_alignment
state_groundedness
```

answer / silence：

```text
grppo_silence_reward=true
grppo_silence_reward_value=0.25
grppo_target_answer_weight=1.0
grppo_target_format_weight=0.0
stepwise_validation_score_key=grppo_target_trajectory_score
```

Judge：

```text
judge.enable=true
judge.backend=gemini
judge.model=gemini-2.5-flash
judge.prompt_version=streamweave_grppo_judge_v1
```

当前代码是两段式 judge：

- 每个 step 跑 process judge，评价 memory / anchor / delta / state。
- 只有 `silence` 或 `answer` supervision step 才跑 answer judge。
- answer judge 只返回 `answer_reward`。

validation 主分：

```text
grppo_target_trajectory_score =
  1.0 * target_answer_reward
  +0.0 * format_mean
```

所以 validation 主要看标注 answer target 的答对率，不把 process judge 混进主分。

## 历史结论

- exp3 到 exp5 做 answer weight sweep：`0.3 -> 0.5 -> 0.7`。

- exp9 系列开始使用 checklist process judge。
- exp9_24 是当前 0516 filter 数据 + SFT 0516 step100 起点的主线。

- REC counting 很弱。
- SSR/CRR 的 evidence gate 不稳。

- OVO 1/8 Total `64.70`，但 OVO full Total 只有 `59.87`。

## 性能经验

历史瓶颈主要在：

```text
old_log_prob
update_actor
```

有效优化方向：

- remove padding。
- fused kernels。
- chunked prefill。
- 关闭不必要 offload。
- 控制 max token 和 micro batch。

注意：

- 不要只看 vLLM 显存利用率，生成速度不是唯一瓶颈。
- `data.gen_batch_size` 控制实际 rollout dataloader batch，不要只看 `train_batch_size`。
- StreamWeave 比 VAGEN 慢的主要原因是更长 prompt / context 和 8B 多模态模型。
- `use_remove_padding=True` 后，Qwen3-VL 的 4-channel MRoPE `position_ids` 不能被错误 squeeze；actor / critic 都要保留 `(4, 1, nnz)` 形态。
- stepwise rollout 会把原始 sample 展开成不等长 turn rows，padding rows 的 `response_mask` 和 `advantages` 必须置零。
- `agg_loss` 要能处理全 padding micro-batch，避免 `0/0 -> NaN`。
- vLLM rollout 采样阶段会 ban Qwen-VL 输入专用的 `image_token_id` / `video_token_id`，通过 `logit_bias=-100` 实现。原因是这些 token 只应该作为 prompt 内的视觉占位符出现；如果被模型生成到 response 里，`old_log_prob` 前向会把 response 中的视觉 token 也计入 image/video token 数量，但没有对应视觉 feature，导致 `Image features and image tokens do not match`。这个设置不改变 prompt 中的视觉 token，也不影响图像特征输入，只禁止非法 response token。

## 当前保留结论

历史调研最终沉淀成以下约束：

1. StreamWeave 训练的是 closed-loop 流式视频 agent，不是离线 QA。
2. 数据要围绕语义漂移、证据不足、延迟回答、多次更新、多 query 同视频构造。
3. RL 标注入口统一为 `query_events`，每个 `answer_event` 都是可评分时间点。
4. 训练和评测必须按 `frames_per_step=5` 的 step 语义运行。
5. 每个 step 输入 `Memory + QA History + Current frames`，每个 step 输出 `<state>/<answer>/<anchor>/<delta>`。
6. 训练必须 no-concat，不能把整条 trajectory 拼成普通 chat transcript。
7. 新 benchmark / 新数据必须保存原始视频、抽帧目录、query time、answer time / interval、选项和答案文本，避免后续无法重构。


### 已淘汰

- `<eta>` 不再使用。
- `frame="N"` 旧字段不作为当前协议。
- VideoXum-StreamQA 作为主训练集的设想已经不再是主线。
- RAFT 主线不再作为当前实现目标。
- 旧 trajectory-level reward 公式不再作为当前 RL 口径。

### 仍保留

- 每个 step 都是 `Memory + QA History + Current frames`。
- `anchor` 保存视觉锚点，`delta` 保存文本语义变化。
- SFT / RL / eval 都必须 closed-loop rollout。
- raw output 和 repaired action 要区分：训练奖励看 raw，环境推进可以用 repair 后结果。
- OVO full 是主评测，StreamingBench 和反幻觉任务是补充评测。

## 数据构造调研

### ReWatch-R1

ReWatch 的流程：

```text
长视频 dense caption
  -> 基于 detailed caption 合成 QA
  -> 过滤 text-only / summary-only 能答的偏置题
  -> 合成 grounded CoT / tool-like reasoning
  -> SFT + RLVR
```

对当前项目的启发：

- 新数据不能只靠题面或常识能答，必须过滤 text-only / summary-only bias。
- 干扰项应来自视频内相似证据或相似事件，不能随便编。
- 如果以后合成长视频推理数据，要保留 dense caption、证据时间、过滤结果和最终 QA 的对应关系。

### OVO-Bench

OVO 的关键中间标注：

```text
query_time
clue_time / evidence_interval
answer_time / clues_reveal_time
question
answer
options
```

任务分组：

```text
Backward: EPM / ASI / HLD
Realtime: STU / OJR / ATR / ACR / OCR / FPD
Forward:  REC / SSR / CRR
```

对当前项目的价值：

- CRR 是 evidence sufficiency / answerability，不是普通内容回答。
- REC / SSR 适合训练 running-count 和 step tracking。
- HLD / CRR 暴露证据不足时的拒答边界。
- OVO full 是当前最重要的 closed-loop 检验。

### Streamo

Streamo-Instruct-465K 的核心是流式状态监督：

```text
<Silence>   当前无关或证据未出现
<Standby>   事件进行中但还不能回答
<Response>  信息足够，输出答案
```

它包含：

```text
narration
event/action caption
event grounding
time-sensitive QA
offline QA
```

对当前项目最直接有用的是 time-sensitive QA：同一个问题在多个时间点有不同答案，可映射成：

```text
query_events[].answer_events[]
```

因此 Streamo 是一问多答、语义漂移、多次更新数据的主要来源。

### CogStream

CogStream 对当前项目的价值不是“问题之间有强逻辑依赖”，而是：

- 同一视频可以聚合多个 query，降低训练推理成本。
- query 多数是实时回答型。
- 可以直接对齐 `query_events` schema。

处理时要注意：

- 没有抽帧目录的视频必须丢掉。
- 同 step 多 query 或多 answer target 必须过滤或拆开。
- 聚合后仍要保证 RL 框架一条视频只推理一次。

### StreamBridge / AURA / MMDuet2

共同点：

- 都强调 interleaved video-text。
- 都把“什么时候说话”作为流式视频理解的一部分。
- 都有主动响应、沉默、延迟回复或多次回复的设计。

对当前项目的取舍：

- 不单独训练 activation model。
- 把回答/沉默作为 RL answer supervision。
- 用 `answer_policy` 和 `answer_events` 表达实时回答、延迟回答和多次更新。
- 同视频多 query 是训练效率方向，但必须避免同 step 冲突。

### Eyes Wide Open / ESTP-Bench

重点：

- ego-streaming proactive QA。
- 每个问题有一个或多个 valid answer intervals。
- 一部分问题是 context-linked，需要依赖历史 QA。

对当前项目的启发：

- 如果后续补“依赖历史问题”的数据，应优先找带 `previous_qa_ids` 或 context-linked 标注的来源。
- 最好保存 valid answer interval，而不是只保存单点 answer_time。
- 训练时可落到 interval 末尾，评测时再看 timeliness。

## Agentic RL 调研

`docs/2026-04-30_agentic_rl_stateful_rollout_literature_report.md` 的核心结论已经进入当前 RL 代码。

StreamWeave 的状态转移是：

```text
s_t = video_progress_t + memory_t + qa_history_t + current_frames_t
prompt_t = render(s_t)
a_t = model(prompt_t)
s_{t+1} = env.step(s_t, a_t)
```

因此当前主线必须坚持：

- fresh prompt per step。
- non-concat step-level training row。
- 一条视频完整 rollout 后再做 reward / credit assignment。
- `group_idx / traj_idx / turn_idx` 保留同视频多 rollout 的结构。
- 同一 sample、同一 turn 的多个 rollout 做组内比较。
- step reward 和 answer reward 分开建模，再合并到 GRPPO advantage。

## 参考方向

### Agent Lightning

可保留观点：

- agent execution 应拆成 transition。
- 不应简单 sequence concatenation with masking。

当前对应实现：

```text
StreamWeaveAgentLoop
AgentLoopOutput
group_idx / traj_idx / turn_idx
```

### VAGEN

可保留观点：

- VLM 多步环境适合 no-concat training。
- 每一步的 prompt 依赖环境状态，不能预先静态拼好。

当前对应实现：

```text
StreamWeaveRLEnv
fresh prompt per step
```

### RAGEN / GUI-Shepherd

可保留观点：

- 长序列 agent 需要过程 reward。
- 不能只靠最终二值 reward。

当前对应实现：

```text
process judge:
  delta_groundedness
  anchor_keyframe
  semantic_alignment
  state_groundedness

answer judge:
  silence / answer supervision
```

### MobileRL

可保留观点：

- 难度过滤和方差过滤有价值。

当前对应实现：

```text
Gemini 4-repeat difficulty cleaning
answer_events[].eval_pass_rate
rl_0516_filter.jsonl
```

### WebAgent-R1

可保留观点：

- 终局二值 reward 可以让 agent 学到行为。

当前取舍：

- StreamWeave 有更丰富 step signal，不应只用终局二值 reward。
- 当前 GRPPO 同时使用 process reward、format reward、note frequency reward 和 answer reward。

## 已放弃或降级的方向

| 方向 | 当前处理 |
|---|---|
| 只做离线 QA SFT | 不够，必须 closed-loop rollout |
| 只奖励最终答案 | 不够，必须 stepwise process reward |
| 单独 activation model | 暂不做，用 answer supervision 训练沉默/回答 |
| 直接拼接整条轨迹训练 | 不做，采用 no-concat stepwise training |
| 只看 OVO 1/8 | 不作为结论，full 才是主结论 |
| 只看 StreamingBench proactive | 不够，还要看 SQA、real、反幻觉 |
| 随机编干扰项 | 不建议，干扰项应来自视频内相似事件或视觉相似证据 |

## 后续可回收方向

后续如果继续扩展数据和 benchmark，优先级如下：

1. 带 valid answer interval 的 streaming QA。
2. 一个视频多个 query，且 query 时间不同。
3. 明确 evidence sufficiency / unable-to-answer 的数据。
4. 需要历史 QA 才能回答的 context-linked 数据。
5. 时间敏感的一问多答数据。
6. 可自动过滤 text-only bias 的合成数据。

这些方向都必须最终落回当前 canonical schema：

```text
video row
  -> query_events[]
    -> answer_events[]
```

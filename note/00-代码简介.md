# 代码简介

本文记录 `streamweave_v5` 当前源码结构和真实运行链路。旧文档中出现的 `<eta>`、`frame="N"`、V3/V4 旧协议和早期 GRPO 入口不再作为当前事实。

## 核心思想

StreamWeave 是一个流式视频理解框架。每个 step 输入：

```text
Memory + QA History + Current frames
```

模型输出：

```xml
<state>...</state>
<answer>...</answer>
<anchor t="..."></anchor>
<delta t="...">...</delta>
```

含义：

- `state`：当前 step 的判断过程和证据摘要，不写入 Memory。
- `answer`：当前应该回答或更新答案时输出；否则保持空。
- `anchor`：视觉锚点，保留关键帧图像。
- `delta`：文本语义变化，压缩锚点之间或窗口之间的状态变化。

核心约束：

- 每次输出必须有且只有一个 `<state>` 和一个 `<answer>`。
- 输出必须以 `<state>`、`<answer>` 开头。
- `<answer>` 后只能出现 `<anchor>` 和 `<delta>`。
- `<anchor>` 只允许 `t` 属性，必须写成成对标签。
- `<delta>` 必须有非空文本和合法时间区间。
- 至少要有一个 observation tag：`anchor` 或 `delta`。

## 整体分层

代码库可以按 7 层理解：

| 层 | 目录 | 作用 |
|---|---|---|
| 核心 runtime | `streamweave/` | 定义 StreamWeave 的协议、prompt、memory、parser、quality、rollout。Eval、SFT、RL 都复用这里。 |
| 模型 backend | `backend/` | 把 Gemini、OpenAI-compatible vLLM、mock backend 统一成 `generate()` 接口。 |
| 评测 | `evaluation/`、`dataset3/` | 跑 OVO、StreamingBench、VideoHallucer、EventHallusion，并把模型输出转成分数和 trace。 |
| 数据合成 | `data_engine/`、`dataset2/` | 上游视频/标注处理，SFT teacher 合成，RL query_events 重构和难度清洗。 |
| SFT 训练 | `SFT/LlamaFactory/` | LLaMAFactory 训练配置、dataset_info、checkpoint、导出模型。 |
| RL 训练 | `RL/streamweave_rl/`、`RL/scripts/`、`RL/configs/` | verl agent-loop 适配、stepwise rollout、reward/judge/advantage、启动脚本。 |
| 产物和记录 | `models/`、`outputs/`、`note/` | 训练模型、评测输出、实验文档。 |

核心原则：`streamweave/` 是协议和状态机的源头；`evaluation/`、`data_engine/sft/`、`RL/streamweave_rl/` 只是把这个状态机接到不同任务上。

## Runtime 链路

核心代码在 `streamweave/`。

关键文件：

```text
streamweave/config.py        配置 dataclass，加载 yaml/json 到 EvalConfig
streamweave/schemas.py       共享数据结构：FrameRef、BenchmarkSample、ModelAction、Transition、RolloutTrace
streamweave/prompts.py       prompt 构造：teacher_synthesis、teacher_eval、production/eval
streamweave/parser.py        XML 解析和严格校验
streamweave/quality.py       时间、格式、open-tail、gap 检查
streamweave/postprocess.py   deterministic repair，eval/RL 推进 memory 时用
streamweave/memory.py        anchor/delta/QA history memory，负责渲染和窗口淘汰
streamweave/policies.py      memory policy：anchor_delta、delta_only、anchor_only、recent_frames
streamweave/env.py           单步环境状态机：build_prompt / evaluate_attempt / commit
streamweave/rollout.py       视频流式 rollout 主循环，eval 主路径
streamweave/frame_store.py   抽帧目录读取、manifest 检查、必要时抽帧
streamweave/video_extract.py 通用抽帧工具
streamweave/trace_io.py      trace 写出辅助
```

主流程：

```text
BenchmarkSample
  -> FrameStore
  -> RolloutRunner
  -> StreamWeaveEnv.build_prompt()
  -> backend.generate()
  -> parser / quality
  -> postprocess repair
  -> memory commit
  -> RolloutTrace
```

`StreamWeaveEnv` 是最核心的状态机：

- `build_prompt()`：把当前 memory、QA history、当前帧窗口渲染成多模态 prompt。
- `evaluate_attempt()`：解析 raw XML，计算 quality，必要时 repair。
- `commit()`：把 repaired action 写入 Memory，包括 anchor、delta、answer。

默认视频设置：

```text
sample_fps = 1.0
frames_per_step = 5
memory_window_seconds = 120.0
```

`frame_count` 是数据标注中的有效视频长度。RL 和清洗脚本都应按 `frame_count` 截断，而不是直接相信帧目录里全部图片。

## Backend 层

模型调用统一在 `backend/`：

```text
backend/base.py       BaseBackend、MockBackend、ContentItem 等接口
backend/gemini.py     Gemini / VertexAI 调用
backend/openai.py     OpenAI-compatible chat completions，本地 vLLM 也走这里
backend/pool.py       多 endpoint round-robin，用于多 vLLM worker
backend/factory.py    根据 config 创建 backend
backend/image_utils.py 图片 resize / 编码辅助
backend/retry.py      重试策略
```

上层代码不要直接写 Gemini/vLLM 细节，应该通过 `backend.factory.create_backend()` 拿统一 backend。这样 eval、SFT teacher、数据清洗可以复用同一套模型调用口径。

## Memory 策略

Memory 保存三类内容：

- anchor：图像锚点。
- delta：文本状态变化。
- QA History：问题和已回答案。

常用 policy：

```text
streamweave / anchor_delta  读 anchor + delta
delta_only                  只读 delta，不读 anchor 图像
anchor_only                 只读 anchor，不读 delta
recent_frames               不读 memory，只看最近帧
no_memory                   不读 memory
```

当前主线默认使用 `streamweave`。

## Eval 链路

评测入口：

```text
evaluation/eval_ovo.py
evaluation/eval_streamingbench.py
evaluation/eval_batch.py
```

重要文件：

```text
evaluation/runner.py                  单进程 eval 统一 runner
evaluation/eval_batch.py              多进程 / 多 endpoint batch eval
evaluation/ovo_adapter.py             OVO-Bench loader / scorer / summary
evaluation/streamingbench_adapter.py  StreamingBench loader / scorer
evaluation/dataset2_adapter.py        dataset2 旧清洗评测适配
evaluation/rollout_metrics.py         anchor/delta/repair/backend attempt 等指标统计
evaluation/run_data_cleaning_3x.py    旧版 3 次推理清洗框架，0516 RL cleaning 借鉴过
```

Eval 使用 `RolloutRunner.run_sample()` 正常跑完整视频流。最终答案是 trace 中最后一个非空、已提交的 `<answer>`。

`eval_repair` 只做一次后处理，不重试模型。raw XML 的质量仍按模型原始输出统计，Memory 推进使用 repaired action。

普通评测链路：

```text
eval_ovo.py / eval_streamingbench.py
  -> evaluation.runner.run_cli()
  -> load_eval_config()
  -> adapter.load_samples()
  -> backend.factory.create_backend()
  -> FrameStore
  -> RolloutRunner.run_sample()
  -> adapter.score_trace()
  -> results.jsonl / summary
```

`configs/` 里的 yaml 决定 benchmark、backend、runtime、memory policy、输出目录等。常见 batch 配置是 OVO full/1of8、Gemini full、Qwen vLLM、StreamingBench real/sqa/proactive。

## SFT 链路

SFT 数据合成代码在：

```text
data_engine/sft/
```

主入口：

```text
data_engine/sft/run_parallel_pipeline.py
data_engine/sft/run_pipeline.py
data_engine/sft/export_llamafactory.py
```

重要文件：

```text
data_engine/sft/sample_sources.py       统一读取 source=frames / ovo / dataset2
data_engine/sft/dataset2_source.py      dataset2 sft.jsonl / qa_list 分支处理
data_engine/sft/rollout_sft.py          teacher rollout 主逻辑
data_engine/sft/constraints.py          SFT-only answer/anchor 约束
data_engine/sft/export_llamafactory.py  导出 ShareGPT
data_engine/sft/filter_sharegpt_hard.py hard filter，过滤长 delta / 重复行
data_engine/sft/check_sharegpt.py       ShareGPT 格式和图片路径检查
data_engine/sft/run_parallel_pipeline.py SQLite queue 并发合成
```

SFT 合成流程：

```text
SamplePlan
  -> group frames
  -> build StreamWeave prompt
  -> Gemini teacher generate
  -> raw XML strict validate
  -> SFT-only constraints
  -> accepted raw XML commit
  -> sample-level answer check
  -> ShareGPT export
```

关键区别：

- SFT 不把 repair 后的结果当 target。
- 只有 raw XML 本身合法的 teacher 输出才进入训练。
- 失败 attempt 只进 debug，不进入最终训练 JSONL。
- ShareGPT export 默认重建 production prompt，而不是直接使用 teacher prompt。

当前 SFT-only 约束主要是：

- 每 step 最多一个 anchor。
- 长时间没有 anchor 时给 teacher 软提醒。
- 检查 `<answer>` 是否该空或非空。
- 样本级检查最终答案是否匹配 GT。

SFT 训练本身不在 `data_engine/sft/`，而在 `SFT/LlamaFactory/`：

```text
SFT/LlamaFactory/data_streamweave/dataset_info.json
SFT/LlamaFactory/configs/train_streamweave_0516_4500.sh
SFT/LlamaFactory/configs/qwen3vl_8b_full_sft_streamweave_0516_4500.yaml
```

`data_engine/sft/` 负责生成训练 JSONL；`SFT/LlamaFactory/` 负责真正训练 Qwen3-VL。

## RL 链路

RL 自有代码在：

```text
RL/streamweave_rl/
```

主链路：

```text
RL/scripts/train_*.sh
  -> python -m verl.trainer.main_ppo
  -> StreamWeaveAgentDataset
  -> StreamWeaveAgentLoop
  -> StreamWeaveRLEnv.reset()/step()
  -> reward / judge / advantage
```

关键文件：

```text
RL/streamweave_rl/dataset.py              读取 RL JSONL，返回 metadata + dummy tensor
RL/streamweave_rl/env.py                  stepwise 环境，映射 query/answer 到 step
RL/streamweave_rl/agent_loop_stepwise.py  verl agent loop，每个 step 生成一条训练 row
RL/streamweave_rl/rewards.py              format / process / answer / target trajectory reward
RL/streamweave_rl/judge.py                Gemini / local judge 调用和 JSON retry
RL/streamweave_rl/judge_prompts.py        process judge / answer judge prompt
RL/streamweave_rl/scorers.py              MCQ / text / OVO scorer
RL/streamweave_rl/advantage.py            自定义 advantage：stepwise GRPPO 等
RL/streamweave_rl/trace.py                rollout/debug trace
RL/streamweave_rl/smoke_test.py           RL adapter smoke
```

RL 数据必须使用当前统一 schema：

```text
query_events[] 是唯一 QA 标注入口
query_events[].answer_type 只能是 mcq 或 text
query_events[].answer_events[] 是 answer reward 触发点
```

RL 环境按主框架把 `query.time` 和 `answer_event.time` 映射到具体 frame，再映射到 step。每个 step 最多一个 query、最多一个 answer target；冲突数据会直接报错。

RL 和 Eval/SFT 的重要区别：

- RL 只读已抽好的帧，不自动抽帧。
- 缺帧、query 被 `max_steps` 截掉、prompt 过长、空 response 都会 abort trajectory。
- 每个 step 都生成一条训练 row，但通过 `group_idx / traj_idx / turn_idx` 保留同一视频 trajectory 结构。
- reward 按 raw XML 计算，Memory 推进使用 repaired action，避免单步坏输出污染整个 rollout。

verl 改动主要在：

```text
RL/verl/verl/trainer/main_ppo.py          注册 StreamWeave optional integration
RL/verl/verl/trainer/ppo/core_algos.py    自定义 advantage registry
RL/verl/verl/trainer/ppo/ray_trainer.py   stepwise batch align、GRPPO filter、metrics、legacy padding
RL/verl/verl/utils/dataset/rl_dataset.py  data.custom_cls 动态加载
```

原则：prompt、memory、reward 行为优先改 `streamweave/` 或 `RL/streamweave_rl/`；只有 batch 对齐、advantage 接入、verl worker plumbing 才改 `RL/verl/`。



```text
models/qwen3vl_sft_0516_step100
```

当前 reward 方向：

- process judge：评估 delta、anchor、semantic alignment、state groundedness。
- answer judge：在 query/answer target 对应 step 判断回答或沉默是否正确。
- format score：约束 XML 协议。
- note frequency：约束 anchor 频率。
- GRPPO 使用 timeline answer supervision。

## 当前风险

- 仓库其他旧文档仍可能含过期协议和旧脚本口径。
- `RL/README.md` 部分内容可能落后于 0516 数据和 exp9_24。
- `RL/verl/` 和 `SFT/LlamaFactory/` 是大型第三方目录，整理主项目文档时默认不展开。
- 同一视频多 query 可以显著降低训练开销，但必须保证同一个 step 内不出现多个 query 或多个 answer target。
- 修改 RL 数据 schema 时必须同步检查 `RL/streamweave_rl/dataset.py` 和 `RL/streamweave_rl/env.py`。

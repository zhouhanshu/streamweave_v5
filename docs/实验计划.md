# StreamWeave V3 实验计划

> 状态：`draft`
> 日期：`2026-04-25`
> 当前目标：围绕新版单步协议，整理推理、SFT、RAFT、RL 和评测五个部分的实验路线。

当前需求文档中的新版协议作为唯一实现基准。每一步输入由 `memory + qa_history + current frames` 组成，模型输出固定为：

```xml
<eta>...</eta>
<answer>...</answer>
<bridge t="...">...</bridge>
<note t="..." frame="..."/>
```

`note` 只表示被选中的视觉帧和时间区间，不保存文字。`bridge` 保存文字和绝对时间戳。`qa_history` 是按时间顺序记录的问题与模型回答输出日志，不做一问一答配对。后续所有数据构造、训练和评测都围绕这套协议展开。

`eta` 统一表示视频内绝对时间戳，单位为秒，不表示距离当前 step end 的相对 delay。

整体实验分成五个部分：推理代码框架、SFT 数据构造和训练、RAFT 的推理和训练、RL 数据构造和训练、实验评测。当前阶段先把这五部分的边界、输入输出和依赖关系写清楚，具体实现逐步补齐。

## 1. 推理代码框架

当前推理框架实现的是新版单步流式协议。每一步输入由 `memory + qa_history + current frames` 组成，输出固定包含 `eta`、`answer`、`bridge` 和 `note`。入口先按 `sample_fps` 把视频抽成 `000000.jpg、000001.jpg...`，运行时只面对这些抽好的帧；`frames_per_step` 控制每一步给模型看的连续帧数。每个 step 调用一次模型，模型同时完成回答判断、ETA 预测、bridge 生成和 note 选帧。

当前 step 的模型输入顺序固定：

```text
[STREAM_AGENT]

=== Memory ===
历史 bridge 文本
历史 note 时间戳 + 未过期 note 图片

=== QA History ===
历史问答日志；role="q" 表示输入问题，role="a" 表示模型过去输出过的非空 answer

=== Current frames ===
当前 group 内所有采样帧，每帧带 frame id 和绝对时间区间
```

`memory.py` 负责构造 Memory。`bridge` 始终保留文字，格式为：

```xml
<bridge t="10.0-15.0">person walks back to the counter</bridge>
```

`note` 只保留视觉帧和时间戳，不保留文字。未超出 `memory_window` 的 note 会把图片包在 `<note>` 标签内放进模型输入：

```xml
<note t="12.0-12.5">
<image/>
</note>
```

超窗后只保留时间位置：

```xml
<note t="12.0-12.5"/>
```

`qa_history` 统一承载问答历史。Q 到达时写入：

```xml
<qa t="244.0" role="q">What did I drop in the bowl? Options: A. meat; B. onion; C. eggplant; D. carrot;</qa>
```

模型回答后，答案作为独立条目写入：

```xml
<qa t="247.3" role="a">C</qa>
```

Q 和 A 按时间顺序排列。程序不做 query-id 配对，也不判断是否存在未回答问题；模型根据完整 QA History、当前帧和 memory 自己决定本步是否输出 answer。

当前帧由 `prompts.py` 中的 prompt builder 追加到输入中。每帧文本标签和图片相邻：

```xml
<frame id="1" t="10.0-10.5">
<image/>
</frame>
<frame id="2" t="10.5-11.0">
<image/>
</frame>
```

模型输出格式固定，`eta` 和 `answer` 永远不能省略：

```xml
<eta>...</eta>
<answer>...</answer>
<bridge t="...">...</bridge>
<note t="..." frame="..."/>
```

本步不输出 answer 时：

```xml
<eta></eta>
<answer></answer>
```

如果模型根据 QA History、memory 和当前帧判断还不应答，`eta` 给出预测的未来应答时间，使用视频内绝对秒级时间戳，`answer` 为空。证据已经出现且模型决定应答时，`eta` 给出当前视频时间戳，`answer` 输出 benchmark 格式答案，例如 `C`。模型输出的 answer 直接写入问答区，不再有 active query 守卫。

`note` 输出只允许引用当前 group 内的 frame id：

```xml
<note t="11.5-12.0" frame="4"/>
```

程序会先在 raw quality 阶段检查 `frame="4"` 和模型写的 `t` 是否匹配真实帧时间。推理模式下，postprocess 会把可修复的 note 时间改成真实 frame 时间，并打印 warning；frame 越界的 note 会被跳过。trace 中同时保留 raw model output、clean/applied output、quality issues 和 repair warnings。后续 teacher 数据合成或 RAFT/RL 打分可以选择不同后处理策略：teacher 可对坏样本做废弃，RAFT/RL reward 则应直接按 raw output 打分，不做模糊修复。

`bridge` 输出必须带绝对时间区间：

```xml
<bridge t="10.0-11.5">person continues slicing eggplant on the cutting board</bridge>
```

普通 bridge 会先在 raw quality 阶段检查是否落在当前 step 时间范围内。推理模式下，postprocess 会对普通 bridge 做当前窗口内的 interval clamp，并打印 warning。若 memory 最新事件是 bridge，则当前输出的第一条 bridge 会按 open-tail 处理：postprocess 会把起点对齐到旧 bridge 起点，用新 bridge 替换旧 bridge，并在必要时修正 end 到当前 step 范围内。raw quality 仍记录原始输出的问题，后续过滤和指标可以按 raw output 或 clean/applied output 分别统计。

当前推理框架已适配 OVO-Bench 全部任务类型（backward / real-time / forward）。OVO forward 任务（REC/SSR/CRR）对每个 `test_info[i]` 使用独立子视频 `{id}_{i}.mp4`，分别打分后汇总。StreamingBench real / sqa 已接入基础选择题评分；proactive 样本已能加载和 rollout，但当前 `score=None`，LLM judge 语义评分待补。

## 2. SFT 数据构造和训练

SFT 的目标是让学生模型学会当前单步协议、关键帧驱动的 note 选择、bridge 压缩、等待策略和最终回答。数据主体使用 `VideoXum-StreamQA`，视频来自 `ActivityNet_Captions`，关键帧监督来自 `VideoXum` 的 `vsum_onehot`。`TimeChat-Online-139K` 第一波子集（10416 条 QA）保留为备用或参考数据，不作为当前主训练数据。

`VideoXum-StreamQA` 的每条 QA 包含 `video_id`、`question_type`（backward/realtime/forward）、`question`、`answer`、`query_time`、`answer_time`、`evidence_interval`、`evidence_frame_ids`、`source_keyframes`、`visual_fact` 和 `verification` 字段。QA 经过 evidence-only / prefix / text-only / summary-only 四类验证后才进入训练池。

每条 QA 先补一个时间标注 `t*`。`t*` 表示答案已经可以被完整支持的最早时间，对应 `answer_time`。这个标注只用于判断回答是否过早，以及计算 ETA；它不作为 note 位置的硬约束。

教师轨迹生成时，把视频按固定窗口流式输入当前推理框架，例如每步 4s 或 8s。每一步包含当前窗口帧、历史 memory 和 qa history。prompt 里额外列出当前窗口内的候选关键帧：

```text
Candidate key frames in current window:
- t=45s, frame=000045.jpg, visual_fact=...
```

教师看到当前窗口的真实图像，同时根据候选关键帧选择 `<note>`。默认要求 `<note>` 优先引用当前窗口内的候选关键帧；当候选关键帧缺失关键信息时，允许从窗口内其他 1fps 帧补选。VideoXum 关键帧提供选帧先验，visual_fact 提供候选帧语义提示，学生训练输入仍只保留推理框架允许的图像、memory 和 qa_history。

教师输出必须沿用当前协议：

```xml
<eta>...</eta>
<answer>...</answer>
<bridge t="...">...</bridge>
<note t="..." frame="..."/>
```

`note` 用来保存对后续推理有价值的视觉帧，优先来自候选关键帧；`bridge` 写当前窗口和历史 memory 的压缩关系；`answer` 只在当前信息足够时输出。`eta_target` 保存绝对时间戳，不做 `current_window_end` 差分：尚未可答时 `eta_target = t*`；已经可答并输出答案时，`eta_target` 为当前窗口结束时间或实际回答写入时间。

过滤分四步。第一步查格式和时间合法性：XML 字段完整，`note t/frame` 属于当前窗口，bridge 时间不乱跳。第二步查回答时机：`current_window_end < t*` 时如果直接给最终答案，删除；`current_window_end >= t*` 后长时间不答，降权或删除。第三步查 trace 质量：judge 检查 note 是否对应可见内容，bridge 是否忠实压缩当前帧和历史 memory，是否写了无根据内容，是否提前泄露答案。第四步查最终答案：和原始答案做语义一致性判断，答案错的删除。

保留下来的轨迹转成 step-level SFT 样本。每个样本保存完整 rollout step，字段包括 `video_id`、`query`、`answer`、`step_start/end`、当前窗口帧、`candidate_frames`、memory 摘要、教师 XML、`t*`、`eta_target`、过滤分数。SFT 样本的输入就是当前推理输入，输出就是通过 teacher 场景质量检查的教师 XML。当前代码里没有独立 `validator.py`，质量检查逻辑应围绕 `action_quality.py` 和后续 teacher postprocessor 组织。

SFT 评估报告 `parser_valid_rate`、`answer_violation_rate`、`note_rate`、`candidate_note_rate`、`avg_bridge_length` 和 `OVO accuracy`。`candidate_note_rate` 统计 note 来自 VideoXum 候选关键帧的比例，用来判断关键帧先验是否真正进入模型行为。

SFT 部分需要新增或改造：`build_sft_data.py` 从 trace 生成 SFT JSONL，`inspect_sft_data.py` 抽样查看输入、输出、候选关键帧、图片引用和 raw quality / postprocess 结果，`eval_sft_rollout.py` 用学生模型跑同一套 rollout 和 metrics。训练脚本根据最终选定的 VLM 框架再定。

## 3. RAFT 的推理和训练

RAFT 放在 SFT 和 RL 之间，用来修正 SFT closed-loop 中暴露出来的错题。这里的 RAFT 指 `Reward-Ranked Fine-Tuning`，流程比较简单：先用 SFT 模型跑一轮验证集或预留训练池，收集答案错误、格式不稳或 trajectory reward 低的 QA，形成 RAFT 题池；再对这些 QA 多次采样完整 rollout，用和 RL 相同的 reward 给每条 trajectory 打分；最后选择高分轨迹转成 SFT 数据，回灌训练一次。

RAFT 使用短 production prompt 和学生模型自己的 memory。每条错题按 8s 窗口完整 rollout，采样 8 条候选 trajectory。每条候选都保存完整 step 序列：

```text
state_t: memory + qa_history + current_frames
action_t: eta + answer + bridge + note
validation_t: format_valid, time_valid, eta_error
trajectory_score: answer_correct, total_notes, reward
```

排序使用同一套 reward：

```text
R_traj =
  gate_format * (
      w_time * mean_t(R_time_valid_t)
    + w_eta  * mean_t(R_eta_t)
    + w_ans  * R_answer_correct
    - lambda_note * total_notes
  )
```

`gate_format`、`R_time_valid_t`、`R_answer_correct`、`R_eta_t` 和 `lambda_note` 的定义与 RL 部分保持一致。RAFT 只做离线采样和筛选，不更新策略。第一版每条错题保留 reward 最高的一条 trajectory；如果最高分轨迹仍然答案错误，可以先跳过这条 QA，避免把错误答案重新写进 SFT。

回灌训练时，top trajectory 的所有合法 step 都进入 SFT。训练输入仍是当前 step 的 `memory + qa_history + current frames`，target 是该 trajectory 中学生自己采样出的 XML。RAFT 数据和原始 SFT 数据混合训练，比例先控制在较小范围，例如 RAFT:SFT 为 1:3 或 1:4，防止模型过度贴合错题分布。

RAFT 的评估看三件事：SFT 错题重跑后的修正率，SFT+RAFT 在 heldout 上的 answer accuracy，parser valid rate 和 note rate 是否保持稳定。如果 RAFT 后正确率提升但 note 数量明显上升，需要调低 top trajectory 中 note 过多样本的权重，或提高 `lambda_note`。

RAFT 部分需要新增 `collect_sft_errors.py`、`sample_raft_candidates.py`、`rank_candidates.py`、`build_raft_data.py` 和 `eval_raft.py`。

## 4. RL 数据构造和训练

RL 的目标是优化 SFT+RAFT 后仍难以直接对齐的 closed-loop 行为，重点是格式稳定、时间合法、答案正确、ETA 预测和 note 成本。RL 在 SFT+RAFT 模型能够稳定完成 rollout 后启动，数据使用独立的 1k 条 TimeChat QA，视频和 QA 尽量不与 SFT/RAFT 数据重叠。

RL 环境沿用当前推理框架。每条 QA 是一个 trajectory，按 8s 窗口完整 rollout。每一步模型接收短 production prompt 下的 `memory + qa_history + current frames`，输出当前 XML 协议。环境解析输出，更新 memory，记录格式校验、时间校验、ETA 偏差、note 数量和最终答案。RL 阶段使用学生模型自己的历史 memory。

每条 QA 采样 8 条完整 rollout，用同一 QA 内的 8 个 trajectory 做 GRPO 组内相对优化，并加 KL 约束到 SFT+RAFT policy，避免格式和语言风格漂移。

### Reward 总公式

```
R_traj =
  gate_format * (
      w_time  * mean_t(R_time_valid_t)
    + w_eta   * mean_t(R_eta_t)
    + w_ans   * R_answer_correct
    + w_mem   * mean_t(R_memory_t)
  )
```

### 各项说明

**`gate_format`**：硬约束。任意 step 无法解析或缺少必要 XML 字段，整条 rollout 记 0 分。

**`R_time_valid_t`**（0/1，step 级）：检查当前 step 所有时间约束，包括 note frame 引用、note 时间、bridge 绝对时间、普通 bridge 的 step 范围和 open-tail bridge 合法性。

**`R_eta_t`**（连续，step 级）：根据绝对时间戳 `eta_pred` 和 `eta_target` 的偏差计算。尚未可答时目标为 `t*`，已经可答并回答时目标为当前窗口结束时间。6s 以内不惩罚，超过 6s 后随偏差增大；预测方向与真值方向相反且偏差较大时加重惩罚。

**`R_answer_correct`**（0/1，trajectory 级）：由 judge 判断最终答案和标准答案是否语义一致。同一条 rollout 的所有 step 共享同一个值。judge 只接收 question、reference answer 和 model answer，结果缓存。

**`R_memory_t`**（连续，step 级）：核心新增项。目标是在语义保真的前提下最小化 memory token 数量。每步计算方式：

```
R_memory_t = w_sem * SemSim(frames_t, memory_t) - lambda_tok * TokenCount(memory_t)
```

- `SemSim(frames_t, memory_t)`：当前 step 原始图片序列与转换后图文序列（当前 step 的 bridge + 有效 note）之间的语义相似度，取值 [0, 1]。打分器使用多模态模型（待定，初步考虑已有 MLLM），输入为原始帧图片和图文序列文本，输出相似度分数。
- `TokenCount(memory_t)`：当前 step memory 的 token 数量（bridge token + note token），归一化到 [0, 1]。
- `lambda_tok`：token 惩罚系数，作为实验超参控制语义与压缩之间的 trade-off。

语义相似度打分器选型待定，初步方向是使用已有 MLLM（如 Qwen3-VL-32B-Instruct）作为 judge，输入原始帧序列和图文 memory，输出语义保真度评分。

答案正确性是 trajectory-level reward，step 级的时间合法、ETA 和 memory 奖励先聚合成 trajectory score，再在同一 QA 的 8 条 rollout 内做归一化。

RL 的主要风险是 reward hacking。每轮训练后必须跑 heldout closed-loop eval，检查 parser valid rate、time valid rate、answer accuracy、ETA MAE、note rate 和平均 bridge 长度。只要格式稳定性下降，先降低学习率、增大 KL 或停止 RL。RL 部分需要新增 `reward.py`、`rollout_env.py`、`collect_rl_rollouts.py` 和训练脚本。

## 5. 实验评测

第一阶段只保留论文最需要的四组实验，不做鲁棒性实验。主评测以 OVO-Bench 为核心，StreamingBench 作为第二阶段补充。所有系统统一使用新版单步协议输出解析器和同一套 closed-loop rollout 评估。

目的：证明 StreamWeave V3 不是被强 recent-window baseline 轻易打败。

### 5.2 Memory 核心消融

只做五个关键 variant：

| Variant | 输入/记忆配置 | 目的 |
|---|---|---|
| no memory | current frames + `qa_history` | 验证长期 memory 是否必要 |
| note-only | current frames + `qa_history` + selected visual notes | 验证视觉证据笔记的贡献 |
| bridge-only | current frames + `qa_history` + textual bridges | 验证跨片段文本桥接的贡献 |
| note+bridge | current frames + `qa_history` + notes + bridges | 验证完整 memory 的主收益 |
| note+bridge gated | gated 接口占位，当前代码等价于 note+bridge；正式实验前需要补 gate | 验证 memory 成本是否可控 |

报告指标：

| Variant | OVO Overall | Real-Time | Backward | Forward | Note Rate | Bridge Tokens | Memory Tokens |
|---|---:|---:|---:|---:|---:|---:|---:|
| no memory | TBD | TBD | TBD | TBD | TBD | TBD | TBD |
| note-only | TBD | TBD | TBD | TBD | TBD | TBD | TBD |
| bridge-only | TBD | TBD | TBD | TBD | TBD | TBD | TBD |

预期：`bridge-only` 更可能提升 backward / forward 问题，`note-only` 更可能提升细粒度视觉证据问题。`note+bridge gated` 目前只是接口占位；只有补完真实 gate 后，才能报告它是否接近完整效果并降低 memory token 成本。

### 5.3 回答时机实验

比较三种协议设置：

| Setting | 输出 | 目的 |
|---|---|---|
| answer-only | 只输出最终答案 | 检查没有时机控制时是否早答 |
| answer + eta | 输出答案和是否继续看 | 验证显式时机建模的收益 |
| answer + eta + early/late penalty | 在训练/筛选中惩罚早答和迟答 | 验证时机 reward 是否真正改善 rollout |

报告指标：

- `Early Answer Rate`
- `Late Answer Rate`
- `No Answer Rate`
- `ETA MAE`
- `Effective Accuracy`
- `Average Answer Delay`

这组实验应放在论文里作为 StreamWeave 与普通 video QA 的核心区别：模型不只是答对，还要知道什么时候能答。

### 5.4 训练阶段消融

只保留四个训练阶段：
